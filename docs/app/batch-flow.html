バッチ処理フロー設計（骨子）

Phase A-4 骨子版（Phase 9-2 で詳細化） 最終更新: 2026-07-06
本書の位置づけ 画像生成・SNS 投稿バッチのアプリケーション内部の処理手順を定める設計書。Phase A-4 時点は骨子版であり、 冪等性・二重投稿防止・投稿ステータス管理などの詳細は Phase 9-2 で詳細化する（セクション 5）。 全体方針・分冊間の整合は親ページ アプリ設計大枠、データモデルは データモデル設計 を参照。 実行基盤（EventBridge Scheduler → 画像生成 Step Functions → 成功時に SNS 投稿 Step Functions を自動起動）は インフラ設計（infra/workflow.html）のとおりで、本書はその器の中で動くアプリの処理を扱う。

1. 前提・不変条件

  • 1 回の画像生成バッチ起動 = 1 投稿（アカウントごと）を不変条件とする（定義の正はアプリ設計大枠 セクション 2）。1 回の生成実行を generation_runs エンティティとして永続化し、SNS 投稿バッチは生成実行を処理対象の単位とする
  • 本書の処理手順（骨格）は全セット共通とする。セットごとに変わりうる箇所は本文中に「セット別拡張ポイント」と明示し、各セットでどの方式を使うかはセット別設計書に記録する（セット別設計書のルール）

2. 画像生成バッチ処理フロー

  1. SET_CODE から batch_sets を解決する（存在しなければ異常終了、is_active=0 ならスキップして正常終了）
  2. 生成実行（generation_runs）レコードを作成する（セット × SCHEDULED_AT で一意 = 冪等性のアンカー。既存レコードがあれば再実行と判断し、未完了分のみ処理する）
  3. 画像を生成する:
    1. 初期実装は 1 起動 = 1 枚生成（投稿用プロンプトは実質 1 件で運用する）
    2. 画像生成 API（Nano Banana Pro）を呼び出し、画像を S3 に保存し、generated_images を生成実行にぶら下げて登録する（プロンプトスナップショット含む）
  4. 正常終了する（成功により SNS 投稿ワークフローが自動起動される）
「既存レコードがあれば再実行と判断し、未完了分のみ処理する」の意味 Step Functions の Retry 等で同一入力（同じ SET_CODE・SCHEDULED_AT）のタスクが再起動された場合、 generation_runs の INSERT が UNIQUE 制約に当たることで「着手済みのスロットの再実行」と検知できる（UNIQUE 制約は二重生成のガードであると同時に再実行検知のセンサーとして働く）。 このときは新しい生成実行を作らず、既存実行にぶら下がる完了済み画像を確認して足りない分だけ生成する （初期実装の 1 枚生成では「画像がなければ生成、あれば何もせず正常終了」の二択。将来の複数枚生成では部分再開として効く）。 画像 1 枚の完了判定基準（DB 登録済み = 完了、S3 orphan 許容、の旧設計基準を踏襲するか）は Phase 9-2 で定義する。
セット別拡張ポイント: プロンプトの決め方・生成枚数 初期実装は prompt_configs の固定プロンプト（実質 1 件）で 1 枚生成する。将来は、 生成 AI にプロンプト自体を考えさせてから画像を生成する方式、複数プロンプトのローテーション選択・カルーセル素材化、 セット別の生成枚数設定などをセットごとに選択可能にする（2026-07-06 ユーザー要望。どの方式を使うかはセット別設計書に記録する）。 方式が増えても、手順 2 の生成実行の枠組み（1 起動 = 1 生成実行 = 1 投稿分のコンテンツ）は変えない。方式の詳細設計は Phase 9 以降で行う。

3. SNS 投稿バッチ処理フロー

処理対象の単位は「投稿」= 生成実行 × 投稿先アカウントとする（画像単位ではない）。

  1. SET_CODE から batch_sets を解決する（画像生成バッチと同じ扱い）
  2. 投稿対象を決定する: 「投稿試行が一度もない生成実行」のうち最も古い 1 件を対象とする
    • 投稿済みかどうかは、生成実行に紐づく投稿レコード（posts）の有無からアカウントごとに導出する（生成実行テーブル自体に投稿済みフラグは持たない。将来の複数アカウントでアカウントごとに状態が異なるため）
    • 一度でも試行（失敗含む）した生成実行は自動では再対象にしない = 再試行なし・欠番許容（初期実装の方針）
    • 通常運用では直前の画像生成バッチの当回分がそのまま対象になる（未試行の生成実行が複数残っている場合の処理件数は Phase 9-2 で定義する）
    • SCHEDULED_AT に依存しないため、インフラの連携契約（SNS 投稿 Step Functions への入力は set_code のみ）を変更せずに済み、手動での単独実行も同じロジックで動く
  3. 対象の生成実行 × セットの有効な投稿先アカウントごとに順次処理する:
    1. 投稿レコード（posts）を作成する（投稿前の事前記録により二重投稿を防止する。旧設計の考え方を踏襲）
    2. 投稿に載せる画像を post_images で紐づける（当面は 1 枚。将来のカルーセルでは生成実行内の複数画像を並び順付きで束ねる）
    3. キャプションテンプレートを適用して投稿本文を組み立てる
    4. S3 Presigned URL を発行し、Instagram Graph API の 2 段階フロー（コンテナ作成 → publish）で投稿する
    5. 結果（成功 / 失敗、プラットフォーム投稿 ID）を投稿レコードに記録する
  4. 投稿失敗は記録・通知に反映して終了する（「失敗」の定義・通知粒度・終了コード方針は Phase 9-2。運用側の対応手順はアプリ運用設計 セクション 4）
セット別拡張ポイント: キャプションの作り方 初期実装は caption_templates の固定文テンプレートを適用する（手順 3-3）。将来は、 画像内容に合わせて生成 AI がキャプションを作る方式などをセットごとに選択可能にする （2026-07-06 ユーザー要望。どの方式を使うかはセット別設計書に記録する）。
将来のセット別失敗時ポリシー（追いつく / 捨てる） 投稿対象の決定（手順 2）の検索条件を切り替えることで、セット別の失敗時ポリシーを実現する想定: 追いつくモード = 失敗した生成実行も再対象に含める / 捨てるモード = 一定より古い未試行分を対象外にする。 詳細は Phase 9 以降（処理単位を「生成実行」とした経緯はアプリ設計大枠 セクション 2）。

4. ワークフロー連携（環境変数の使い方）

環境変数の受け渡し契約（キー名）は infra/workflow.html セクション 5、業務上の意味付けの定義はアプリ設計大枠 セクション 5 で確定済み。本書のフローでの使い方は以下のとおり。

  • SET_CODE — 両バッチが受け取り、batch_sets の解決に使用する（各フローの手順 1）
  • SCHEDULED_AT — 画像生成バッチのみが受け取り、生成実行の冪等性キーに使用する（セクション 2 の手順 2）。SNS 投稿バッチは受け取らず、投稿対象の決定にも使用しない
  • BATCH_SIZE_LIMIT 等のアプリ用環境変数 — 要否含め Phase 9-2 で定義する

5. Phase 9-2 で詳細化する項目

  • 冪等性・二重投稿防止の具体設計（投稿ステータス遷移、published_unconfirmed 相当の要否）
  • 未完了の生成実行の再開処理（画像 1 枚の完了判定基準を含む）
  • 未試行の生成実行が複数残っている場合の処理件数
  • 画像生成 API が複数画像を返却した場合の処理
  • 「失敗」の定義・通知粒度・終了コード方針
  • BATCH_SIZE_LIMIT 等のアプリ用環境変数の要否
