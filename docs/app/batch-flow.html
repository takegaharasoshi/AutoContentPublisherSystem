バッチ処理フロー設計

Phase 9-2 で詳細化済み・Phase 10-1 で最終 Fix 最終更新: 2026-07-15
本書の位置づけ 画像生成・SNS 投稿バッチのアプリケーション内部の処理手順を定める設計書。Phase A-4 の骨子版を Phase 9-2 で詳細化した（冪等性・二重投稿防止・投稿ステータス管理・失敗時の終了コード方針を確定）。 全体方針・分冊間の整合は親ページ アプリ設計大枠、データモデルは データモデル設計 を参照。 実行基盤（EventBridge Scheduler → 画像生成 Step Functions → 成功時に SNS 投稿 Step Functions を自動起動）は インフラ設計（infra/workflow.html）のとおりで、本書はその器の中で動くアプリの処理を扱う。 本書で確定した内容のデータモデルへの反映は セクション 5 に整理する（Phase 9-3 の入力とする）。

1. 前提・不変条件

  • 1 回の画像生成バッチ起動 = 1 投稿（アカウントごと）を不変条件とする（定義の正はアプリ設計大枠 セクション 2）。1 回の生成実行を generation_runs エンティティとして永続化し、SNS 投稿バッチは生成実行を処理対象の単位とする
  • 本書の処理手順（骨格）は全セット共通とする。セットごとに変わりうる箇所は本文中に「セット別拡張ポイント」と明示し、各セットでどの方式を使うかはセット別設計書に記録する（セット別設計書のルール）
  • 実行ログ記録（共通）: 両バッチとも、SET_CODE から batch_sets を解決した直後（set_id が確定した時点。生成実行の解決や投稿対象の決定より前）に batch_execution_logs へ実行ログを記録する（記録手順は下記を参照）。SET_CODE の解決に失敗した場合（batch_sets が存在しない）は set_id が確定できないため記録は行わず、既存の Step Functions 失敗アラームで検知する。正常終了時は status='succeeded'、異常終了時は status='failed' に更新する。このログ記録自体の INSERT/UPDATE が失敗した場合はバッチ全体を終了コード 1 とする（記録なしでの続行は運用上のリスクが高いため）
  • 起動経路は Step Functions 経由に限る（Phase 10-1 でインフラ構築の知見を反映）: 手動での再実行を含め、バッチは必ず Step Functions（先頭ステートの DB 準備確認 WaitForDbReady 経由）で起動する。バッチアプリ自身は DB 接続リトライを持たない設計（infra/workflow.html セクション 2）のため、ECS RunTask の直接実行は Aurora 自動一時停止中に失敗しうる。この前提により EXECUTION_ARN は常に設定される（手動 RunTask 時の空値は考慮しない）
  • DB アクセスの共通部品: DB 接続には共通ライブラリ shared/acps_shared を使用する（connect() はリトライなし・autocommit OFF。commit は呼び出し側の責務）。本書の INSERT-or-fetch 等の手順は明示的な commit を前提とする
Phase 9-5 レビューでの確定: batch_execution_logs の INSERT/UPDATE 手順（Step Functions Retry 対応） batch_execution_logs は UNIQUE (execution_arn, batch_type) を持つため、Step Functions Retry で同一実行内に ECS タスクが再起動された場合、素朴な INSERT は一意制約違反になる。generation_runs（セクション 2 手順 2）と同じ INSERT-or-fetch パターンを適用する。
  • INSERT 成功 → 新規の実行ログ（attempt_count=1、status='running'）
  • UNIQUE 制約違反 → 既存行を (execution_arn, batch_type) で取得し、attempt_count をインクリメントして status='running' に UPDATE する（前回試行時点の error_message 等は上書きされる。試行ごとの履歴は保持せず、直近の試行状態のみを保持する設計とする）

2. 画像生成バッチ処理フロー

システムとしての対応範囲と初期運用の切り分け システムは1 つの prompt_config から複数画像を保存することと1 セットに複数 prompt_config を持たせることの両方を処理できるようにする（初期実装からこの前提で作る。旧設計の「2 枚目以降は破棄」は採用しない）。 一方、初期運用は 1 セットにつき有効な prompt_config を 1 件のみ設定し、そのプロンプト文言自体を「1 枚だけ生成される」よう作り込む運用ルールとする（セット追加手順は アプリ運用設計 セクション 2.1）。 「1 起動 = 1 プロンプト = 生成画像 1 枚 = 1 投稿 = 投稿画像 1 枚」は設定値・運用ルールで実現する結果であり、システムの実装上の制約ではない。
  1. SET_CODE から batch_sets を解決する。存在しなければ異常終了（終了コード 1）。is_active=0 ならスキップして正常終了（終了コード 0）
  2. 生成実行（generation_runs）を解決する（set_id × SCHEDULED_AT の UNIQUE 制約に対する INSERT を試みる）
    • INSERT 成功 → 新規の生成実行（今回のスケジュール枠の初回実行）
    • UNIQUE 制約違反 → 既存の生成実行を取得する（Step Functions Retry によるタスク再起動、または重複トリガーと判断）。手順 4 で未完了分のみ処理する
  3. 対象の prompt_configs（is_active=1）を全件取得する。0 件なら設定ミスとして異常終了（終了コード 1）（投稿すべき画像を 1 つも作れないため）。1 件以上ある場合はすべてを手順 4 で処理する（初期運用は 1 件を想定するが、複数件でも正しく処理できる）
  4. 各 prompt_config ごとに順次処理する。1 件の失敗は他の prompt_config の処理をブロックしない（例外を捕捉し次の prompt_config へ進む。全体の成否は手順 5 で判定する）
    1. 完了判定: (generation_run_id, prompt_config_id) の組み合わせで generated_images レコードの有無を確認する。存在すれば当該 prompt_config は完了済みとしてスキップする（Step Functions Retry によるタスク再起動時に、画像生成 API を再度呼び出さないため）
    2. 未完了の場合、セットに割り当てられた生成方式（セクション 2.1 の契約に従う strategy モジュール）を実行し、生成された画像をすべて受け取る。0 件ならこの prompt_config の生成失敗としてログ出力し、次の prompt_config へ進む（画像生成 API 呼び出し自体の一時的失敗は Step Functions Retry に委ねる）
    3. 受け取った画像はすべて保存する（1 件でも複数件でも同じ経路で処理する。旧設計の「2 枚目以降は破棄」は採用しない）
    4. 各画像を JPEG に変換し（Instagram 投稿要件に合わせるため。生成 API が PNG を返す場合は変換する）、S3 に保存する
    5. 画像ごとに generated_images にレコードを登録する（S3 キー、prompt_config_id、同一 (generation_run_id, prompt_config_id) 内での並び順を表す output_index、プロンプト本文・パラメータのスナップショットを含む）。1 枚ごとの DB 登録が成功した時点でその画像を完了とする。S3 保存後に DB 登録が失敗した場合は S3 に orphan が残るが許容する
  5. 全 prompt_config の処理後、終了コードを決定して終了する: すべての prompt_config が完了すれば終了コード 0（成功により SNS 投稿ワークフローが自動起動される）。1 つでも未完了が残れば終了コード 1（Step Functions Retry / Catch に委ねる。SNS 投稿バッチの失敗判定方針（3.4）と同じパターン）
再開処理の簡略化に関する既知の割り切り 完了判定は「(generation_run_id, prompt_config_id) に 1 件でも generated_images が存在するか」で行う。API 呼び出しが複数画像のうち一部だけ DB 登録できず失敗した場合でも、再実行時は「1 件存在する = このプロンプトは完了済み」としてスキップし、残りの画像は生成し直さない。厳密な過不足チェックより再開処理の単純さを優先した割り切りであり、問題が顕在化した場合に見直す。

2.1 セット別拡張ポイント: 生成方式（strategy モジュール構造）

決定（2026-07-12 合意・Phase 10-1 反映）: 生成ロジックの隔離方針 将来の多段生成 AI チェーン（プロンプト生成 AI → 画像生成 AI → 品質向上 AI 等）でセットごとに生成ロジックが複雑化しても、 アプリ資材（コンテナ・CI/CD パイプライン）はサービス単位の共有を維持する。そのために image-batch 内部を次の 3 要素で構成する。
  • 共通骨格: セクション 2 の手順 1〜3・5 と手順 4 の枠組み（完了判定 = 冪等性のアンカー、S3 保存、DB 登録）。方式によらず共通
  • 生成方式 strategy モジュール: generators/<方式名>.py（1 方式 1 ファイル）
  • 方式名レジストリ: 方式名 → モジュールの対応表。新方式の追加は「新規ファイル + レジストリ 1 行」に閉じ、既存方式のファイルには触れない
方式の割当は DB のセット設定（方式名の設定値）で行い、コードに set_code の分岐は書かない。 共有部品（共通骨格・API クライアント）の変更は CI/CD パイプラインの pytest ゲート（infra/cicd.html）で守る。

生成方式の契約（方式インターフェース）

  • 方式はセット設定（プロンプト設定等）を受け取り、最終的な生成画像（1 枚以上）を返す
  • 冪等性（完了判定）・S3 保存・DB 登録は共通骨格の責務であり、方式は関与しない
  • 特定の画像生成 API に固有のリクエスト・レスポンス処理やパラメータは方式側（generators/<方式名>.py と方式別設計書）に閉じる。共通設計書・共通骨格には特定 API 名を書かない

方式カタログ

生成方式の設計は本書に書き足さず、docs/app/generators/<方式名>.html に 1 方式 1 本で分冊する（セットが本採用した時点で作成。試行段階では書かない。ルールは アプリ設計大枠 セクション 1.1）。本書には以下のカタログのみを置き、方式の追加・廃止はこの表の行の追加・更新で管理する。廃止した方式は行を削除せず、ステータスを「廃止」にして履歴として残す。

方式名概要ステータス設計書
gpt-image-single（仮称・実装時に確定） 固定プロンプトを ChatGPT Images 2.0（OpenAI、API モデル名 gpt-image-2）に渡す単発生成。初期スコープの方式 計画中（Phase 10 で実装） 本採用時に作成
セット別拡張ポイント: プロンプトの決め方 初期運用は 1 セット 1 prompt_config（固定プロンプト、1 枚生成となるようプロンプト文言で制御）とする。将来の 生成 AI にプロンプト自体を考えさせてから画像を生成する方式などは、新しい生成方式（generators/ の strategy モジュール）として追加し方式カタログに登録する（2026-07-06 ユーザー要望を 2026-07-12 の strategy 構造合意で整理）。 1 セットに複数 prompt_config を有効化してのローテーション選択・カルーセル素材化などは、運用ルールの変更（有効化する prompt_config 件数・プロンプト文言）で実現できる。 いずれの場合も手順 2 の生成実行の枠組み（1 起動 = 1 生成実行 = 1 投稿分のコンテンツ）は変えない。どのセットがどの方式を使うかはセット別設計書に記録する。

3. SNS 投稿バッチ処理フロー

処理対象の単位は「投稿」= 生成実行 × 投稿先アカウントとする（画像単位ではない）。

3.1 投稿対象の決定

  1. SET_CODE から batch_sets を解決する（画像生成バッチと同じ扱い）
  2. 投稿対象を決定する: 「投稿試行が一度もない生成実行」のうち最も古い 1 件を対象とする
    • 投稿済みかどうかは、生成実行に紐づく投稿レコード（posts）の有無からアカウントごとに導出する（生成実行テーブル自体に投稿済みフラグは持たない。将来の複数アカウントでアカウントごとに状態が異なるため）
    • 一度でも試行（失敗含む）した生成実行は自動では再対象にしない = 再試行なし・欠番許容（初期実装の方針）
    • SCHEDULED_AT に依存しないため、インフラの連携契約（SNS 投稿 Step Functions への入力は set_code のみ）を変更せずに済み、手動での単独実行も同じロジックで動く
Phase 9-3 での厳密化: 「投稿試行が一度もない」の DB クエリ定義 上記の判定条件は、Step Functions Retry によるタスク再起動時に同じ生成実行を再選択して 3.2 の復旧ロジックへつなげる必要があるため、 「posts 行が 1 つもない」ではなく「有効なアカウントのうち少なくとも 1 つが終端状態（success / failed / published_unconfirmed）に達していない」を正確な条件とする。 SQL・インデックス設計は data-model.html セクション 4.4 を参照。
決定: 未試行の生成実行が複数残っている場合の処理件数 1 回の実行では常に最古の 1 件のみ処理し、BATCH_SIZE_LIMIT のような処理件数の上限設定は設けない。 通常運用では直前の画像生成バッチの当回分がそのまま対象になるため 1 件で足り、SNS 投稿が連続して失敗しバックログが積み上がった場合も、 「1 回の欠番は致命的でない」という初期方針（A-1 壁打ち記録 セクション 4）のもと、複数件をまとめて追いつかせる複雑な制御はあえて持たない。 未処理のまま残った古い生成実行は、将来のセット別失敗時ポリシー（追いつくモード）導入時に扱う。

3.2 posts の状態遷移（冪等性・二重投稿防止）

posts は生成実行 × アカウントで一意（UNIQUE 制約 (generation_run_id, sns_account_id)）とし、attempt_number 相当のカラムは持たない。自動再試行を行わない初期方針のため、1 つの投稿対象について作られる行は常に 1 行であり、その行の状態遷移だけで進行状況を表現する。

状態意味遷移条件
pending投稿処理に着手した（予約）投稿対象と決定した直後、API 呼び出し前に INSERT する
container_createdコンテナ作成 API まで成功（platform_container_id 設定済み）コンテナ作成 API 成功時に更新
success投稿完了（platform_post_id 設定済み）パブリッシュ API が成功時に更新
failed投稿未完了で終了。自動では再対象にしない終端状態API が明確に失敗を返した場合に更新
published_unconfirmed投稿済みの可能性を否定できない終端状態。要手動確認重複 publish 応答、またはパブリッシュ要求送信後のタイムアウト・接続断
Step Functions Task Retry（同一実行内の再起動）での復旧 SNS 投稿 ECS タスクが異常終了し Step Functions の Retry（最大 2 回、30 秒間隔。infra/workflow.html セクション 6）でタスク全体が再起動された場合、対象生成実行に紐づく有効アカウントごとに posts の状態を確認し、以下のとおり分岐する。
  • 行が存在しない、または pending（platform_container_id 未設定）→ 最初から実行（コンテナ作成 → パブリッシュ）
  • container_created（platform_container_id 設定済み、platform_post_id 未設定）→ パブリッシュから再開（同じ container_id を使い、新しいコンテナは作らない）
  • success → スキップ（このアカウントへの投稿は完了済み）
  • failed / published_unconfirmed → スキップ（同一実行内の Step Functions Retry でも自動では再試行しない。終端状態の扱いは全実行を通じて一貫させる）
この分岐により、Step Functions Retry が働いても既に成功・終端状態のアカウントへの無駄な API 呼び出しは発生しない。

3.3 処理手順

  1. 対象の生成実行に対し、セットの有効な投稿先アカウント（sns_accounts）ごとに順次処理する。1 アカウントの失敗は他アカウントの処理をブロックしない（例外を捕捉し次のアカウントへ進む）
  2. posts の状態（3.2）に基づき、未完了のアカウントについて処理を進める:
    1. posts レコードが存在しなければ pending で作成する（INSERT-or-skip。3.2 の復旧分岐により、既存行がある場合は作成せずその状態から再開する。投稿前の事前記録により二重投稿を防止する）
    2. 初期構築の仕様: 生成実行に紐づく画像のうち最初の 1 枚のみを post_images で紐づける（選択順序は generated_images.id 昇順 = 最も早く生成された 1 枚。1 セット複数 prompt_config や 1 プロンプト複数画像により生成実行が複数の画像を保存していても、投稿するのは常に 1 枚とする）。将来のカルーセル対応では、生成実行内の複数画像を並び順付きで束ねる処理に拡張する
    3. キャプションテンプレートを適用して投稿本文を組み立てる
    4. S3 Presigned URL を発行し（有効期限 1 時間）、Instagram Graph API のコンテナ作成 API（POST /{ig-user-id}/media）を呼び出す。取得した platform_container_id を即座に保存し、posts.status を container_created に更新する
    5. パブリッシュ API（POST /{ig-user-id}/media_publish）を呼び出す
      • 成功 → platform_post_id を保存し status='success' に更新
      • 明確な失敗（投稿未完了と判定できる応答）→ status='failed' に更新
      • 重複 publish 応答、またはタイムアウト・接続断で結果不明 → status='published_unconfirmed' に更新
  3. 全アカウントの処理後、3.4 の方針に基づき終了コードを決定して終了する
送信元 IP に関する制約（Phase 10-1 でインフラ構築の知見を反映） Fargate タスクは固定 IP を持たない（NAT なし・タスクごとにパブリック IP が変わる構成。infra/workflow.html セクション 3）。 送信元 IP の許可リスト登録を必要とする API・プラットフォームとは、現行構成のままでは組み合わせられない。

3.4 「失敗」の定義・通知粒度・終了コード方針

  • 失敗の定義: 対象の生成実行に対し、有効な投稿先アカウントのうち1 つでも success にならなかった場合（failed または published_unconfirmed で終わったアカウントが 1 つでもある場合）を失敗とする
  • 終了コード: 全アカウントが success なら 0。1 つでも非 success が残るなら 1
  • 終了コード 1 により Step Functions の既存 Retry（最大 2 回）が働くが、3.2 の状態遷移によって success / 終端状態のアカウントはスキップされるため、Retry によって無駄な API 呼び出しが増えることはない。Retry を使い切ってもなお非 success が残る場合は Catch → Fail 状態に遷移し、既存の CloudWatch Alarm（ExecutionsFailed）で通知される
  • 通知粒度: 初期スコープでは個別アカウント・個別失敗理由ごとのカスタムメトリクスは持たない。既存の Step Functions 失敗アラーム 1 本のみで通知し、詳細な原因（どのアカウント・どの段階で失敗したか）は CloudWatch Logs と posts テーブル（status / api_response）で確認する運用とする（確認手順は アプリ運用設計 セクション 4.1）
  • published_unconfirmed も「対応が必要な状態」として終了コード 1 の対象に含める（success と明確に区別し、運用者に手動確認を促すため）
セット別拡張ポイント: キャプションの作り方 初期実装は caption_templates の固定文テンプレートを適用する（3.3 の手順 3）。将来は、 画像内容に合わせて生成 AI がキャプションを作る方式などをセットごとに選択可能にする （2026-07-06 ユーザー要望。どの方式を使うかはセット別設計書に記録する）。
将来のセット別失敗時ポリシー（追いつく / 捨てる） 投稿対象の決定（3.1）の検索条件を切り替えることで、セット別の失敗時ポリシーを実現する想定: 追いつくモード = 失敗した生成実行も再対象に含める / 捨てるモード = 一定より古い未試行分を対象外にする。 詳細は Phase 9 以降（処理単位を「生成実行」とした経緯はアプリ設計大枠 セクション 2）。

4. ワークフロー連携（環境変数の使い方）

環境変数の受け渡し契約（キー名）は infra/workflow.html セクション 5、業務上の意味付けの定義はアプリ設計大枠 セクション 5 で確定済み。本書のフローでの使い方は以下のとおり。

  • SET_CODE — 両バッチが受け取り、batch_sets の解決に使用する（各フローの手順 1）
  • SCHEDULED_AT — 画像生成バッチのみが受け取り、生成実行の冪等性キーに使用する（セクション 2 の手順 2）。SNS 投稿バッチは受け取らず、投稿対象の決定にも使用しない
  • BATCH_SIZE_LIMIT 等のバッチ件数制御用環境変数は追加しない（確定）。SNS 投稿バッチは常に生成実行 1 件のみを処理する方針（3.1）のため件数上限の概念が不要であり、画像生成バッチも有効プロンプト 1 件・1 枚生成の初期スコープでは総量制御が不要と判断した。将来、生成実行のバックログが恒常的に蓄積する場合や画像生成 API のクォータ制約が顕在化した場合に導入を再検討する

5. Phase 9-2 の決定事項と Phase 9-3 への申し送り

本書の詳細化に伴いデータモデル（data-model.html）側で確定・反映が必要な事項を以下に整理する。

  • posts.status の値集合: pending / container_created / success / failed / published_unconfirmed（3.2）
  • posts の UNIQUE 制約は (generation_run_id, sns_account_id) のみ。旧設計にあった attempt_number 相当のカラムは不要（自動再試行を行わない方針のため。3.2）
  • posts に platform_container_id / platform_post_id / api_response 等、2 段階フローの中間状態を保持するカラムが必要（3.3）
  • generated_images の完了判定は (generation_run_id, prompt_config_id) の組み合わせでの存在確認により行う。完了状態を表す専用カラムは不要（2）
  • generated_images に output_index（同一 generation_run_id × prompt_config_id 内での並び順）カラムが必要。画像生成 API が返す image part はすべて保存する方針のため（旧設計の「2 枚目以降は破棄」は不採用。2）
  • post_images は初期構築では生成実行内の先頭画像（generated_images.id 昇順で最初の 1 枚）のみを登録する。生成実行が保存する画像枚数によらず、投稿は常に 1 枚（3.3）
  • prompt_configs の is_active=1 件数は 0 件をエラー、1 件以上はすべてループ処理する方針を確定（DB 制約の変更は不要。処理ロジックの決定事項として記録。2）
  • BATCH_SIZE_LIMIT 等のアプリ用環境変数は追加しない（4）

参考資料: _archive/batch.md（冪等性・再試行・投稿ステータス管理の旧設計。post_records の attempt 方式や BATCH_SIZE_LIMIT によるバックログ処理は、自動再試行なし・生成実行単位への簡素化に伴い初期スコープでは採用しない。現役の設計書ではない）
