3. 主キーと冪等性キー・セット境界の整合性

主キーと FK 参照の方針（旧設計を踏襲） 各テーブルの主キーは AUTO_INCREMENT の id とし、他テーブルからの FK 参照は基本的に id に対して行う。 generation_runs の「set_id × scheduled_at」は FK の参照先ではなく、冪等性を保証する UNIQUE 制約としてのみ使用する。 scheduled_at に入る値は EventBridge Scheduler が渡すスケジュール上の実行予定時刻（UTC）であり、 リトライ・再実行でも同じスロットなら同じ値になるため、スロットの安定した識別子として機能する。
セット境界の整合性を保証する複合 FK（旧設計のパターンを踏襲・拡張） generated_images・posts は、参照先テーブルの主キー id に加えて set_id を非正規化して保持し、 (set_id, 対象カラム) → 参照先テーブル(set_id, id) の複合 FK を張る（参照先テーブル側は UNIQUE (set_id, id) を持つ）。 これにより、異なるセットに属する generation_runs / prompt_configs / sns_accounts / caption_templates を 誤って組み合わせて参照することを DB レベルで防止する。この保証はセット境界までとし、 「post_images.generated_image_id が同一 generation_run に属すること」のような、より粒度の細かい整合性はアプリ層の責務とする （複合 FK をここまで深くすると generation_run_id の非正規化が連鎖し複雑さに見合わないと判断。セクション 4.6 に既知の割り切りとして明記）。

4. テーブル定義（詳細）

全テーブルの正確な型・制約・インデックスは database/V001__initial_schema.sql を正とする。以下は各テーブルの設計意図。

4.1 batch_sets

  • set_code（UNIQUE）が S3 キー・Secret 名・EventBridge Scheduler 入力・環境変数 SET_CODE に使われる外部識別子。DB 内部の FK 参照はすべて id で行う
  • is_active=0 でセット廃止を表現する（レコードは削除しない。operation.html セクション 2.2）

4.2 prompt_configs

  • is_active=1 の全件が画像生成バッチのループ対象（batch-flow.html セクション 2）。初期運用は 1 セット 1 件のみ有効化する運用ルールで「1 起動 1 枚生成」を実現する
  • UNIQUE (set_id, id) は generated_images からの複合 FK の参照先として使用する

4.3 caption_templates

決定: 紐づけ単位はセット、選定ルールは id 昇順の 1 件 キャプションテンプレートの紐づけ単位はセットとする（プロンプト設定単位にはしない。画像の内容とキャプションは独立に差し替えたいケースが多いため）。 SNS 投稿バッチは投稿本文の組み立て時（batch-flow.html セクション 3.3 手順 3）に、対象セットの caption_templates から is_active=1 のうち id 昇順で先頭の 1 件を選択して適用する。 複数の is_active=1 レコードを持つこと自体はテーブル制約上許容する（将来のローテーション・バリエーション拡張の余地）が、 初期運用では 1 セットにつき is_active=1 は 1 件のみ設定する運用ルールとする（prompt_configs の運用方針と対称）。 is_active=1 が 0 件の場合はキャプションなし（空文字）で投稿処理を継続する（画像投稿の続行を優先し、キャプション欠落は投稿失敗の理由にしない）。
  • template_text はキャプション本文とハッシュタグを含む 1 本の固定文（Instagram の投稿本文はキャプション欄 1 つのため分割しない）
  • posts.caption_template_id + posts.caption_text_snapshot で「どのテンプレートを」「どんな本文で」投稿したかを追跡する（generated_images のプロンプトスナップショットと同じ設計思想）

4.4 generation_runs

  • UNIQUE (set_id, scheduled_at) が冪等性キー。画像生成バッチはこの制約への INSERT 成否で「新規スロット」か「同一スロットへの再実行（Step Functions Retry・重複トリガー）」かを判定する（batch-flow.html セクション 2 手順 2）
  • 投稿済みかどうかのフラグ・生成完了かどうかのフラグはどちらも持たない。前者は posts の有無、後者は generated_images の有無から導出する（状態を単一のテーブルに集約せず、実体（画像・投稿）の存在自体を正とする設計）
決定: SNS 投稿バッチの投稿対象決定クエリの正確な定義

batch-flow.html セクション 3.1 の「投稿試行が一度もない生成実行」は、データモデルとしては以下のとおり厳密化する。 ある generation_run が投稿対象の候補（actionable）となるのは、 有効な（is_active=1）投稿先アカウントのうち少なくとも 1 つについて、posts 行が存在しないか、終端状態（success / failed / published_unconfirmed）に達していない場合である。 対象は actionable な generation_run のうち scheduled_at が最も古い 1 件。

SELECT gr.id
FROM generation_runs gr
WHERE gr.set_id = :set_id
  AND EXISTS (
    SELECT 1 FROM sns_accounts sa
    WHERE sa.set_id = gr.set_id AND sa.is_active = 1
      AND NOT EXISTS (
        SELECT 1 FROM posts p
        WHERE p.generation_run_id = gr.id
          AND p.sns_account_id = sa.id
          AND p.status IN ('success','failed','published_unconfirmed')
      )
  )
ORDER BY gr.scheduled_at ASC, gr.id ASC
LIMIT 1;

この定義により: (1) 新規の generation_run（posts なし）は当然対象になる (2) Step Functions Retry による同一実行内のタスク再起動時も同じ生成実行が再選択され、batch-flow.html セクション 3.2 の復旧ロジック（pending / container_created からの再開）が機能する (3) 異常終了（Step Functions のリトライ上限到達等）で pending / container_created のまま残った generation_run は、後続の別実行でも自動的に再選択され続ける（stale データとして後続投稿をブロックし得る問題の検知・補正手順は operation.html セクション 4.3 で確定した）。 「一度でも試行（失敗含む）した生成実行は自動では再対象にしない」（batch-flow.html セクション 3.1）は、「有効な全アカウントが終端状態に達した」場合を指すと解釈する。

インデックスは generation_runs.uq_generation_runs_set_id (set_id, id)（scheduled_at 順スキャン）と posts.uq_posts_generation_run_account (generation_run_id, sns_account_id)（NOT EXISTS 部分検索）の左端プレフィックスで十分であり、追加インデックスは設けない。

4.5 generated_images

  • output_index（NEW）: 同一 (generation_run_id, prompt_config_id) 内での並び順。画像生成 API が返す image part をすべて保存する方針（旧設計の「2 枚目以降は破棄」は不採用）に対応する
  • 完了判定は UNIQUE (generation_run_id, prompt_config_id, output_index) の左端プレフィックス (generation_run_id, prompt_config_id) の存在確認で行う。専用の完了フラグは持たない（batch-flow.html セクション 2 手順 4.1）
  • set_id は非正規化カラム（セクション 3 の複合 FK 用）。generation_run_id と prompt_config_id はそれぞれ (set_id, ...) の複合 FK で参照する

4.6 posts / post_images

  • posts の UNIQUE (generation_run_id, sns_account_id) により 1 投稿対象につき常に 1 行。状態遷移は batch-flow.html セクション 3.2 の 5 状態（pending / container_created / success / failed / published_unconfirmed）をそのまま status ENUM に反映する
  • caption_template_id（nullable FK）+ caption_text_snapshot: posts 行は本文組み立て前（pending 作成時）に INSERT されるため、当初は両方 NULL。本文組み立て時（セクション 3.3 手順 3）に UPDATE で設定する
  • post_images は 1 投稿 : N 画像（将来のカルーセル）に対応する中間テーブル。初期構築では生成実行内の先頭画像（generated_images.id 昇順で最初の 1 枚）のみを display_order=0 で 1 行登録する（batch-flow.html セクション 3.3 手順 2）
既知の割り切り: post_images.generated_image_id の生成実行境界チェックはアプリ層 post_images.generated_image_id が「投稿対象の generation_run に属する画像であること」は DB の FK では強制しない（セクション 3 の複合 FK 方針の対象外）。 アプリケーションは常に対象 generation_run_id に紐づく generated_images からのみ選択するため実害は小さいと判断し、 generation_run_id をさらに非正規化して複合 FK を重ねる複雑さは避けた。

4.7 batch_execution_logs との役割分担

決定: generation_runs と batch_execution_logs は独立（相互 FK なし） batch_execution_logs は「バッチプログラムの 1 回の起動（Step Functions 実行 / ECS タスク）」を記録する運用ログであり、 generation_runs は「1 回の起動が作った投稿単位のコンテンツ」を表す業務データである。 両者は目的が異なるため FK で結合しない。画像生成バッチは SET_CODE 解決前や generation_run 確定前に失敗する場合もあり、 batch_execution_logs は generation_run の有無に関わらず必ず 1 件記録される必要があるため、独立させる方が実装が単純になる。 どの実行がどの generation_run を作ったかを調べたい場合は、started_at と generation_runs.created_at の近接時刻で運用者が突き合わせる（通常は 1 実行 : 1 生成実行の関係になるため実務上問題ない）。

status / attempt_count の INSERT/UPDATE 手順（Step Functions Retry による再起動時に UNIQUE (execution_arn, batch_type) と衝突しないための INSERT-or-fetch パターン）は batch-flow.html セクション 1 を参照（Phase 9-5 レビューで確定）。

なお、画像生成と SNS 投稿は別ステートマシンのため、1 つの論理実行（画像生成 → 連鎖起動された SNS 投稿）に対応する 2 行は execution_arn が異なる。子ワークフローは親と同一の実行名で起動される（infra/workflow.html セクション 3）ため、両バッチの実行ログを突き合わせる場合は execution_arn 末尾の実行名部分で相関を取る（Phase 10-1 でインフラ構築の知見を反映）。

5. S3 キー設計

決定: S3 キー命名規約 
images/{set_code}/{YYYYMMDD}/{generation_run_id}/{prompt_config_id}_{output_index}.jpg
  • {set_code} は SET_CODE 環境変数（batch_sets.set_code）
  • {YYYYMMDD} は generation_runs.scheduled_at（UTC）の日付部分。日付プレフィックスにより S3 コンソール・CLI での目視確認や範囲指定操作がしやすくなる
  • {generation_run_id} / {prompt_config_id} / {output_index} は generated_images の該当カラムそのもの。この 3 つの組み合わせは UNIQUE (generation_run_id, prompt_config_id, output_index) と一致するため、S3 キーの一意性は DB の一意性制約と対応する
  • 拡張子は常に .jpg（generated_images.file_format は将来の形式追加に備えた記録用カラムであり、初期運用は JPEG 固定）
決定: セット廃止時に「残す」データの範囲（設計課題リストの解消） development-plan.md 設計課題リストで持ち越していた「セット廃止時に残す対象が DB レコードか S3 実体も含むか」を以下のとおり確定する。 「残す」の対象は DB レコード（generated_images のメタ情報・posts の投稿履歴等）のみであり、S3 実体（画像ファイル）はインフラ設計の 30 日ライフサイクルポリシー（infra/stacks.html セクション 3.1）により作成から 30 日で自動削除される。 セット廃止後もメタ情報・投稿履歴は DB に残るため、「いつ・どんなプロンプトで・どのアカウントに投稿したか」という記録は失われないが、 画像ファイル自体の永続保存が必要になった場合（収益化後の資産化等）は、S3 ライフサイクル対象外の別バケット・別プレフィックスへの複製を将来検討する（初期実装では対応しない）。

6. 時刻の取り扱い

  • DB の全 DATETIME カラムはUTC で保存する。MySQL の DATETIME 型はタイムゾーン情報を持たないため、アプリケーション側で UTC に変換してから格納する
  • generation_runs.scheduled_at の正規化: EventBridge Scheduler が渡す環境変数 SCHEDULED_AT は ISO 8601（YYYY-MM-DDTHH:MM:SSZ、秒精度、Z 接尾辞で UTC を示す）。アプリケーションは末尾の Z を除去し YYYY-MM-DD HH:MM:SS 形式にして DATETIME カラムへ格納する（タイムゾーン変換は不要。すでに UTC のため）
  • CURRENT_TIMESTAMP は DB セッションのタイムゾーンに依存するため、アプリケーションは DB 接続直後に SET time_zone = '+00:00' を実行する
  • Aurora 側でも可能な限り Parameter Group で time_zone='UTC' を設定し、手動 DDL / Query Editor 実行時も UTC 前提を維持する
  • 冪等性キー（generation_runs の UNIQUE 制約）の比較・判定もすべて UTC ベースで行う。1 日複数回実行は scheduled_at の時刻部分で区別される
  • 表示時のタイムゾーン変換（UTC → Asia/Tokyo 等）はアプリケーション層の責務とする（初期実装では管理画面がないため当面発生しない）

7. 文字コード・照合順序

  • 文字コード: utf8mb4
  • 照合順序: utf8mb4_unicode_ci（旧設計の決定を踏襲。絵文字を含むキャプション本文・プロンプト本文を正しく扱える）
