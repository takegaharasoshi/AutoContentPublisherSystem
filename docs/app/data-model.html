2. エンティティ一覧

エンティティ役割旧設計からの変更点・備考
batch_sets セット定義（1 セット = 1 テーマ）。外部識別子は set_code 将来のセット別設定（失敗時ポリシー・生成枚数・プロンプト / キャプションの生成方式等）のカラム拡張余地を持たせる。max_post_retries は初期スコープ外（再試行なし）
prompt_configs 画像生成プロンプト設定（セット : 設定 = 1 : N） 旧設計を踏襲（スナップショット方式含む）。初期は投稿用プロンプト実質 1 件で運用。将来の「AI にプロンプトを考えさせる方式」では、固定プロンプトの代わりにメタプロンプト等を保持する拡張を想定（セット別拡張ポイント。詳細は Phase 9 以降）
generation_runs新規 生成実行（1 回の画像生成バッチ起動で 1 レコード）。投稿の単位であり、冪等性のアンカー（セット × SCHEDULED_AT で一意） A-2 レビューで新設。「1 生成実行 = 1 投稿（アカウントごと）」を構造で保証し、将来のカルーセルの「画像の束」もこの単位で定まる
caption_templates新規 投稿キャプション・ハッシュタグの固定文テンプレート A-1 で新規に必要と判明。紐づけ単位（セット / プロンプト設定 / アカウント）は Phase 9 で決定。AI 生成キャプションは将来のセット別方式（セット別拡張ポイント）
generated_images 生成画像のメタ情報（S3 キー・プロンプトスナップショット等）。generation_runs の子（1 実行 : N 画像） 生成実行にぶら下げる構造へ変更。複数画像の保存を許容（旧設計の「2 枚目以降破棄」を廃止。具体構造は Phase 9）
sns_accounts 投稿先アカウント（セット : アカウント = 1 : N を維持） 旧設計を踏襲。当面は Instagram 1 アカウントだが、将来の他プラットフォーム展開のため 1 : N 構造を残す
posts新規 投稿 1 件（生成実行 × アカウント。キャプション・投稿ステータス・プラットフォーム投稿 ID を持つ） 旧 post_records（画像 × アカウント × 試行）を「投稿」単位に再編。生成実行が「投稿されたか」は本テーブルの有無からアカウントごとに導出する（generation_runs にフラグは持たない）
post_images新規 投稿と画像の中間テーブル（投稿に載せる画像の選択と表示順） 1 投稿 : N 画像（将来のカルーセル）に対応。当面の運用は 1 投稿 1 画像
batch_execution_logs バッチ実行ログ（運用ログ） 旧設計を踏襲。業務データである generation_runs とは役割を分ける（役割分担の整理は Phase 9）
post_insights将来 投稿の効果測定データ（いいね数・リーチ等） 将来のインサイト収集バッチ（第 3 のバッチ）用。posts に紐づける前提で拡張余地のみ確保

3. 主キーと冪等性キーの分離

方針（旧設計を踏襲） 各テーブルの主キーは AUTO_INCREMENT の id とし、他テーブルからの FK 参照はすべて id に対して行う。 generation_runs の「セット × SCHEDULED_AT」は FK の参照先ではなく、冪等性を保証する UNIQUE 制約としてのみ使用する （DATETIME 型が他テーブルのキーに伝播することはない）。 SCHEDULED_AT に入る値は EventBridge Scheduler が渡すスケジュール上の実行予定時刻（UTC）であり、 リトライ・再実行でも同じスロットなら同じ値になるため、スロットの安定した識別子として機能する。

4. Phase 9-3 で詳細化する項目

4.1 テーブル定義・DDL・マイグレーション方針 Phase 9 で詳細化

全テーブルのカラム定義・制約・インデックスを設計し、database/ に本スキーマの DDL を作成する。DDL のバージョン管理・マイグレーション方針もあわせて定義する。generation_runs と batch_execution_logs の役割分担、キャプションテンプレートの紐づけ単位と変数展開の要否もここで確定する。

4.2 S3 キー設計 Phase 9 で詳細化

複数画像（1 生成実行 : N 画像）に対応した S3 キー体系を設計する。あわせて、インフラ設計の S3 30 日ライフサイクルと「セット廃止時もデータは残す」方針の整理（「残す」対象が DB レコードか S3 実体も含むか）を確定する（development-plan.md 設計課題リストに記録済み）。

4.3 時刻の取り扱い Phase 9 で詳細化

SCHEDULED_AT の ISO 8601 文字列から DATETIME への正規化ルール（UTC 固定・秒精度等）、およびテーブル全般のタイムゾーン方針を定義する。
