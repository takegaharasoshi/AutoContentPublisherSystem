2. 処理フロー概要

実行基盤（EventBridge Scheduler → 画像生成 Step Functions → 成功時に SNS 投稿 Step Functions を自動起動）はインフラ設計（infra/workflow.html）のとおり。ここではバッチアプリケーション内部の処理手順の骨子を定める。

不変条件: 1 回の画像生成バッチ起動 = 1 投稿（アカウントごと） 1 回の生成実行が「1 投稿分のコンテンツ」を作る。この単位を generation_runs（生成実行）エンティティとして永続化し、 投稿バッチは生成実行を処理対象の単位とする。将来のカルーセル投稿でも「1 実行で生成された複数画像 = 1 投稿に束ねる画像群」となり、この不変条件は変わらない。 将来の多プラットフォーム展開時は、同一の生成実行がアカウント数分の投稿レコードになる（内容は 1 つ）。

2.1 画像生成バッチ

  1. SET_CODE から batch_sets を解決する（存在しなければ異常終了、is_active=0 ならスキップして正常終了）
  2. 生成実行（generation_runs）レコードを作成する（セット × SCHEDULED_AT で一意 = 冪等性のアンカー。既存レコードがあれば再実行と判断し、未完了分のみ処理する）
  3. 画像を生成する:
    1. 初期実装は 1 起動 = 1 枚生成（投稿用プロンプトは実質 1 件で運用する。複数プロンプトの活用 — カルーセル素材・ローテーション選択等 — は将来拡張）
    2. 画像生成 API（Nano Banana Pro）を呼び出し、画像を S3 に保存し、generated_images を生成実行にぶら下げて登録する（プロンプトスナップショット含む）
  4. 正常終了する（成功により SNS 投稿ワークフローが自動起動される）
「既存レコードがあれば再実行と判断し、未完了分のみ処理する」の意味 Step Functions の Retry 等で同一入力（同じ SET_CODE・SCHEDULED_AT）のタスクが再起動された場合、 generation_runs の INSERT が UNIQUE 制約に当たることで「着手済みのスロットの再実行」と検知できる（UNIQUE 制約は二重生成のガードであると同時に再実行検知のセンサーとして働く）。 このときは新しい生成実行を作らず、既存実行にぶら下がる完了済み画像を確認して足りない分だけ生成する （初期実装の 1 枚生成では「画像がなければ生成、あれば何もせず正常終了」の二択。将来の複数枚生成では部分再開として効く）。 画像 1 枚の完了判定基準（DB 登録済み = 完了、S3 orphan 許容、の旧設計基準を踏襲するか）は Phase 9-2 で定義する。

2.2 SNS 投稿バッチ

処理対象の単位は「投稿」= 生成実行 × 投稿先アカウントとする（画像単位ではない）。

  1. SET_CODE から batch_sets を解決する（画像生成バッチと同じ扱い）
  2. 投稿対象を決定する: 「投稿試行が一度もない生成実行」のうち最も古い 1 件を対象とする
    • 投稿済みかどうかは、生成実行に紐づく投稿レコード（posts）の有無からアカウントごとに導出する（生成実行テーブル自体に投稿済みフラグは持たない。将来の複数アカウントでアカウントごとに状態が異なるため）
    • 一度でも試行（失敗含む）した生成実行は自動では再対象にしない = 再試行なし・欠番許容（A-1 の初期方針）
    • 通常運用では直前の画像生成バッチの当回分がそのまま対象になる（未試行の生成実行が複数残っている場合の処理件数は Phase 9 で定義する）
    • SCHEDULED_AT に依存しないため、インフラの連携契約（SNS 投稿 Step Functions への入力は set_code のみ）を変更せずに済み、手動での単独実行も同じロジックで動く
  3. 対象の生成実行 × セットの有効な投稿先アカウントごとに順次処理する:
    1. 投稿レコード（posts）を作成する（投稿前の事前記録により二重投稿を防止する。旧設計の考え方を踏襲）
    2. 投稿に載せる画像を post_images で紐づける（当面は 1 枚。将来のカルーセルでは生成実行内の複数画像を並び順付きで束ねる）
    3. キャプションテンプレートを適用して投稿本文を組み立てる
    4. S3 Presigned URL を発行し、Instagram Graph API の 2 段階フロー（コンテナ作成 → publish）で投稿する
    5. 結果（成功 / 失敗、プラットフォーム投稿 ID）を投稿レコードに記録する
  4. 投稿失敗は記録・通知に反映して終了する（「失敗」の定義・通知粒度・終了コード方針は Phase 9）
処理単位を「生成実行」とした経緯（旧設計・A-2 初版からの変更） 旧設計および A-2 初版は「画像」単位で投稿対象を検索していたが、A-2 レビューでのユーザー指摘 （投稿バッチの処理単位は「投稿」であるべき。画像単位では将来のカルーセルで画像の束ね方が定まらない）を受け、 「1 生成実行 = 1 投稿（アカウントごと）」を不変条件とし、投稿対象の単位を生成実行に変更した（2026-07-06）。 将来のセット別ポリシーは、この検索条件の切り替えで実現する想定 （追いつくモード = 失敗した生成実行も再対象に含める / 捨てるモード = 一定より古い未試行分を対象外にする）。詳細は Phase 9。

3. データモデル概要

主要エンティティと関係を以下に定める。カラム定義・制約・インデックス・DDL は Phase 9-3 で設計する。

エンティティ役割旧設計からの変更点・備考
batch_sets セット定義（1 セット = 1 テーマ）。外部識別子は set_code 将来のセット別設定（失敗時ポリシー・生成枚数等）のカラム拡張余地を持たせる。max_post_retries は初期スコープ外（再試行なし）
prompt_configs 画像生成プロンプト設定（セット : 設定 = 1 : N） 旧設計を踏襲（スナップショット方式含む）。初期は投稿用プロンプト実質 1 件で運用
generation_runs新規 生成実行（1 回の画像生成バッチ起動で 1 レコード）。投稿の単位であり、冪等性のアンカー（セット × SCHEDULED_AT で一意） A-2 レビューで新設。「1 生成実行 = 1 投稿（アカウントごと）」を構造で保証し、将来のカルーセルの「画像の束」もこの単位で定まる
caption_templates新規 投稿キャプション・ハッシュタグの固定文テンプレート A-1 で新規に必要と判明。紐づけ単位（セット / プロンプト設定 / アカウント）は Phase 9 で決定。AI 生成キャプションは将来拡張
generated_images 生成画像のメタ情報（S3 キー・プロンプトスナップショット等）。generation_runs の子（1 実行 : N 画像） 生成実行にぶら下げる構造へ変更。複数画像の保存を許容（旧設計の「2 枚目以降破棄」を廃止。具体構造は Phase 9）
sns_accounts 投稿先アカウント（セット : アカウント = 1 : N を維持） 旧設計を踏襲。当面は Instagram 1 アカウントだが、将来の他プラットフォーム展開のため 1 : N 構造を残す
posts新規 投稿 1 件（生成実行 × アカウント。キャプション・投稿ステータス・プラットフォーム投稿 ID を持つ） 旧 post_records（画像 × アカウント × 試行）を「投稿」単位に再編。生成実行が「投稿されたか」は本テーブルの有無からアカウントごとに導出する（generation_runs にフラグは持たない）
post_images新規 投稿と画像の中間テーブル（投稿に載せる画像の選択と表示順） 1 投稿 : N 画像（将来のカルーセル）に対応。当面の運用は 1 投稿 1 画像
batch_execution_logs バッチ実行ログ（運用ログ） 旧設計を踏襲。業務データである generation_runs とは役割を分ける（役割分担の整理は Phase 9）
post_insights将来 投稿の効果測定データ（いいね数・リーチ等） 将来のインサイト収集バッチ（第 3 のバッチ）用。posts に紐づける前提で拡張余地のみ確保
主キーと冪等性キーの分離 各テーブルの主キーは AUTO_INCREMENT の id とし、他テーブルからの FK 参照はすべて id に対して行う（旧設計の方針を踏襲）。 generation_runs の「セット × SCHEDULED_AT」は FK の参照先ではなく、冪等性を保証する UNIQUE 制約としてのみ使用する （DATETIME 型が他テーブルのキーに伝播することはない）。 SCHEDULED_AT に入る値は EventBridge Scheduler が渡すスケジュール上の実行予定時刻（UTC）であり、 リトライ・再実行でも同じスロットなら同じ値になるため、スロットの安定した識別子として機能する。 ISO 8601 文字列から DATETIME への正規化ルール（UTC 固定・秒精度等）は Phase 9-3 で定義する。

4. 運用方針の骨子

運用方針（初期実装）
セット追加 (1) Secrets Manager に SNS 認証 Secret 作成 → (2) DB に batch_sets / prompt_configs / caption_templates / sns_accounts を登録 → (3) CDK に EventBridge Scheduler を追加して deploy
セット廃止・入れ替え（スクラップ&ビルド） is_active=0 化 + Scheduler 削除。データ（生成画像・投稿履歴）は残す
プロンプト・キャプション変更 DB 直接操作（Query Editor 等）。管理画面は将来拡張
投稿失敗時 メール通知を受けて Step Functions 実行履歴・CloudWatch Logs・DB を確認する。自動再試行はしない（欠番許容）。必要な場合のみ SNS 投稿ワークフローを手動起動する（未試行分が投稿される）
Instagram トークン更新 60 日失効前に手動リフレッシュ（旧設計の運用を踏襲。手順詳細は Phase 9-4）
効果測定 Instagram アプリ・インサイト画面で手動確認。自動収集は将来拡張

5. ワークフロー連携・Secret 規約の意味付け（Phase A スコープ分の確定）

インフラ設計で「アプリ設計で確定する」とされていた項目のうち、大枠レベルで確定できるものを以下に定める。

項目意味付け
SET_CODE batch_sets.set_code（人が定義する安定した外部識別子）。EventBridge Scheduler の入力で指定し、両バッチが受け取って内部 ID に解決する
SCHEDULED_AT スケジュール実行時刻（UTC、Scheduler の <aws.scheduler.scheduled-time> 由来）。生成実行（generation_runs）の冪等性キー（セット × SCHEDULED_AT で一意）であり、1 日複数回実行は DATETIME 粒度で区別される。SNS 投稿バッチは受け取らず、投稿対象の決定にも使用しない（セクション 2.2 の検索方式のため。現行の連携契約の変更は不要）
SNS Secret 規約の構成要素 acps/{env}/{set_code}/sns/{platform}/{account_code} のうち、set_code = batch_sets.set_code、account_code = sns_accounts.account_code（作成後変更不可、Secret 名導出専用）。将来の 1 セット多プラットフォーム展開とも {platform} の分岐で整合する
アプリ用環境変数（BATCH_SIZE_LIMIT 等） 要否含め Phase 9 で定義する（投稿対象が「最古の未試行生成実行 1 件」となるため、旧設計ほどの総量制御が必要か再検討する）

6. Phase 9 への持ち越し（大枠と詳細の線引き）

本書（および A-4 の大枠設計書）で扱うのはここまでとし、以下は Phase 9 で設計する。

  • 9-2 バッチ処理フロー詳細: 冪等性・二重投稿防止の具体設計（投稿ステータス遷移、published_unconfirmed 相当の要否）、未完了の生成実行の再開処理、未試行の生成実行が複数残っている場合の処理件数、複数画像返却時の処理、「失敗」の定義・通知粒度・終了コード方針、BATCH_SIZE_LIMIT 等の要否
  • 9-3 データモデル詳細: 全テーブルのカラム定義・制約・インデックス・DDL（generation_runs と batch_execution_logs の役割分担の確認を含む）、S3 キー設計（複数画像対応）、キャプションテンプレートの紐づけ単位と変数展開の要否、時刻の取り扱い、マイグレーション方針
  • 9-4 アプリ運用詳細: セット追加・トークン更新の具体手順、失敗時の確認・手動補正手順、stale データの扱い
  • 将来拡張の具体化（実装時期は未定）: 失敗時ポリシーのセット別モード（追いつく / 捨てる）、生成枚数・プロンプト選択のセット別設定（複数プロンプトのカルーセル素材化・ローテーション等）、AI キャプション生成、カルーセル投稿処理、インサイト収集バッチ、リール・動画、承認フロー、管理画面
