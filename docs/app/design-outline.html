1.1 セット別設計書のルール（複数セット前提の 2 層構造）

経緯（2026-07-06） A-4 計画レビューで「スクラップ&ビルドでセットが増えていく前提で、共通・上位の設計は変更せず、 セットが増えるたびに設計書が 1 本増えるような管理しやすい体系にしたい」というユーザー要望を受け、 共通設計書とセット別設計書の 2 層構造を定めた（要望の記録は A-1 壁打ち記録の追記を参照）。
  • 2 層構造: 共通設計書（本書 + 3 分冊）は全セット共通の骨格を定め、セット追加では変更しない。セット追加で増える設計書は docs/app/sets/<set_code>.html の 1 本のみ
  • セット別設計書の標準記載項目: テーマ、投稿先アカウント構成、スケジュール（投稿頻度）、プロンプトの作り方（固定 / AI 生成等）、キャプションの作り方（テンプレート / AI 生成等）、失敗時ポリシー、共通フローからの差分・特記事項
  • 拡張ポイント方式: セットごとに変わりうる箇所は、共通分冊側に「セット別拡張ポイント」として明示する（batch-flow.html セクション 2・3）。新しい方式が共通の骨格に影響する場合は、共通分冊に拡張ポイントを追記した上で、セット別設計書には選択と差分のみを書く（同じ情報を 2 箇所に書かない）
  • 実データとの役割分担: プロンプト文言・スケジュール時刻等の実データの正は DB / CDK コード。セット別設計書には意図・方針・方式の選択を書く
  • 廃止時の扱い: セット廃止（スクラップ&ビルド）では設計書を削除せず、廃止の旨を付けて履歴として残す（手順は operation.html セクション 2.2）
  • インフラは増えない: インフラ設計書・CDK スタックはセット追加で増えない・変わらない。スタックは機能単位であり、セットは「既存 ImageBatchStack 内の Scheduler 1 件 + Secrets Manager の Secret + DB レコード」として表現される（operation.html セクション 2.1）
  • 作成タイミング: セット別設計書の実ファイル・雛形は初セット追加時に作成する（A-4 時点ではルールのみ定義）

2. 全体方針: 処理単位の不変条件

不変条件: 1 回の画像生成バッチ起動 = 1 投稿（アカウントごと） 1 回の生成実行が「1 投稿分のコンテンツ」を作る。この単位を generation_runs（生成実行）エンティティとして永続化し、 投稿バッチは生成実行を処理対象の単位とする。将来のカルーセル投稿でも「1 実行で生成された複数画像 = 1 投稿に束ねる画像群」となり、この不変条件は変わらない。 将来の多プラットフォーム展開時は、同一の生成実行がアカウント数分の投稿レコードになる（内容は 1 つ）。
処理単位を「生成実行」とした経緯（旧設計・A-2 初版からの変更） 旧設計および A-2 初版は「画像」単位で投稿対象を検索していたが、A-2 レビューでのユーザー指摘 （投稿バッチの処理単位は「投稿」であるべき。画像単位では将来のカルーセルで画像の束ね方が定まらない）を受け、 「1 生成実行 = 1 投稿（アカウントごと）」を不変条件とし、投稿対象の単位を生成実行に変更した（2026-07-06）。 将来のセット別失敗時ポリシー（追いつく / 捨てる）は投稿対象の検索条件の切り替えで実現する想定（batch-flow.html セクション 3）。

この不変条件のもとでの処理手順の骨子（画像生成: セット解決 → 生成実行の作成 = 冪等性のアンカー → 生成・S3 保存・DB 登録 / SNS 投稿: 最古の未試行生成実行を対象にアカウントごとに投稿）は、分冊 バッチ処理フロー設計 に定める。

3. データモデル概要

「1 生成実行 = 1 投稿」を構造で保証するため、generation_runs（生成実行）を中心に、 posts（投稿 = 生成実行 × アカウント）・post_images（1 投稿 : N 画像の中間テーブル）・caption_templates（キャプションテンプレート）を新設し、 batch_sets / prompt_configs / generated_images / sns_accounts / batch_execution_logs は旧設計の構造を踏襲・再編する。 post_insights（効果測定）は将来拡張の余地のみ確保する。 ER 図・エンティティ一覧・主キーと冪等性キーの分離方針は、分冊 データモデル設計 に定める。 データモデルは全セット共通であり、セット間の違いはレコード・設定カラムで表現する。

4. 運用方針の骨子

セット追加（セット別設計書の作成 → Secret 作成 → DB 登録 → Scheduler 追加 deploy）、セット廃止（is_active=0 + Scheduler 削除。データと設計書は残す）、 プロンプト / キャプションの DB 直接運用、投稿失敗時の対応（自動再試行なし・通知のみ・必要時のみ手動起動）、 Instagram トークンの手動更新、効果測定（当面は手動確認）の各方針は、分冊 アプリ運用設計 に定める。

5. ワークフロー連携・Secret 規約の意味付け（Phase A スコープ分の確定）

インフラ設計で「アプリ設計で確定する」とされていた項目のうち、大枠レベルで確定できるものを以下に定める。 フロー上の使い方は batch-flow.html セクション 4、Secret の運用は operation.html セクション 5 を参照。

項目意味付け
SET_CODE batch_sets.set_code（人が定義する安定した外部識別子）。EventBridge Scheduler の入力で指定し、両バッチが受け取って内部 ID に解決する
SCHEDULED_AT スケジュール実行時刻（UTC、Scheduler の <aws.scheduler.scheduled-time> 由来）。生成実行（generation_runs）の冪等性キー（セット × SCHEDULED_AT で一意）であり、1 日複数回実行は DATETIME 粒度で区別される。SNS 投稿バッチは受け取らず、投稿対象の決定にも使用しない（batch-flow.html セクション 3 の検索方式のため。現行の連携契約の変更は不要）
SNS Secret 規約の構成要素 acps/{env}/{set_code}/sns/{platform}/{account_code} のうち、set_code = batch_sets.set_code、account_code = sns_accounts.account_code（作成後変更不可、Secret 名導出専用）。将来の 1 セット多プラットフォーム展開とも {platform} の分岐で整合する
アプリ用環境変数（BATCH_SIZE_LIMIT 等） 要否含め Phase 9 で定義する（投稿対象が「最古の未試行生成実行 1 件」となるため、旧設計ほどの総量制御が必要か再検討する）
