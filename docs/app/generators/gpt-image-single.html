方式設計書: gpt-image-single

Fix（13-2 で作成） 方式名（レジストリキー）: gpt-image-single 最終更新: 2026-07-23
本書の位置づけ 生成方式 gpt-image-single の方式固有仕様（使用 API・パラメータ・画像変換）を記録する。全方式共通の契約（方式インターフェース）と方式カタログは batch-flow.html セクション 2.1、分冊ルール（1 方式 1 本・本採用時に作成）は design-outline.html セクション 1.1 を参照。本書は実装（services/image-batch/app/generators/gpt_image_single.py）を正として記述している。

1. 方式概要

固定プロンプトを OpenAI Images API（ChatGPT Images 2.0、API モデル名 gpt-image-2）に渡し、1 回の API 呼び出しで画像を生成する単発生成方式。初期スコープの方式であり、初セット fantasy-animals-1 が使用する（Phase 11-4 で実装、13-1 で定常運用開始 = 本採用）。

項目値
方式名（レジストリキー = batch_sets.generator_name の値）gpt-image-single
実装ファイルservices/image-batch/app/generators/gpt_image_single.py
レジストリ登録services/image-batch/app/generators/__init__.py の REGISTRY（dict へ手動で 1 行登録）
使用 APIOpenAI Images API（client.images.generate）
出力形式JPEG バイト列のリスト（list[bytes]）

2. 契約の実装

Phase 14-4 の契約一般化に伴う予定変更 方式契約は Phase 14-4 で「投稿可能な最終形式の最終メディア（本体 + メタ情報）を返す」に一般化された（batch-flow.html セクション 2.1）。 本方式のシグネチャ・戻り値型はこの一般化に合わせて 14-7 で更新予定（JPEG を返す処理内容は変わらず、戻り値へのメタ情報付与のみ）。以下は現行実装（更新前）を正とした記載。

共通の契約（batch-flow.html「生成方式の契約」）を次のシグネチャで実装する。

generate(prompt_config: PromptConfig) -> list[bytes]

処理順序: ① API キーを Secret から取得 → ② OpenAI クライアント生成 → ③ Images API 呼び出しで PNG を取得 → ④ 各 PNG を JPEG に変換 → ⑤ JPEG バイト列のリストを返す。冪等性（完了判定）・S3 保存・DB 登録は契約どおり共通骨格の責務であり、本方式は関与しない。

3. 入力

入力は PromptConfig（prompt_configs テーブルの 1 行。data-model.html）のみ。使用するフィールドは以下。

フィールド使い方
prompt_textAPI の prompt にそのまま渡す
parameters（JSON 文字列）API パラメータ。下表の 4 キーすべて必須
negative_prompt本方式では未使用（共通骨格が generated_media〔14-5 のリネーム前は generated_images〕のスナップショットに記録するのみ）

parameters の JSON キー仕様:

キー型渡し先例（fantasy-animals-1 の実値）
model文字列images.generate(model=...)gpt-image-2
size文字列images.generate(size=...)1024x1024
quality文字列images.generate(quality=...)high
n整数images.generate(n=...)（生成枚数）1
パラメータのバリデーションは行わない 4 キーはデフォルト値なしの必須参照であり、キー欠落・parameters が NULL・不正 JSON の場合は例外（KeyError / TypeError / JSONDecodeError）がそのまま送出される。例外は共通骨格が捕捉し、その prompt_config をスキップしてバッチ全体は失敗終了（フェイルラウド）となる（セクション 6）。各セットの実値はセット別設計書と DB（prompt_configs.parameters）で管理する。

4. 認証情報（API キー）

環境変数 API_SECRET_ARN（infra/workflow.html セクション 5.1 の Task Definition 静的設定。実体は acps/{env}/image/api-key）を方式モジュールが直接読み、acps_shared.get_secret_string で取得した SecretString（JSON {"api_key": "..."}）をパースして API キーを得る。取得は generate() 呼び出しごとに行う（キャッシュなし。初期運用は 1 実行 1 prompt_config のため実害なし）。

決定（Phase 11-4）: Secret は方式モジュールが自分で取得する API Secret は共通骨格から引数で渡さず、方式モジュールが環境変数から自分で取得する。共通骨格（main.py / processing.py / レジストリ型 GeneratorFn = Callable[[PromptConfig], list[bytes]]）に特定 API の認証都合を持ち込まないための設計であり、契約「特定 API 固有の処理は方式側に閉じる」に沿う。将来プロバイダごとの Secret 分離が必要になった場合の扱いは設計課題リスト（2026-07-15、2 プロバイダ併用時）を参照。

5. API 呼び出し

項目値
クライアントopenai.OpenAI(api_key=..., timeout=300.0)（タイムアウト 300 秒固定。画像生成は数分かかるため）
呼び出しclient.images.generate(model, prompt, size, quality, n)
レスポンス処理response.data[].b64_json を base64 デコードして PNG バイト列を取得（n に応じて複数枚）
リトライ方式側では行わない（openai SDK のデフォルト挙動に依存。明示指定なし）

6. 画像処理（PNG → JPEG 変換）

API が返す PNG を Pillow で JPEG に変換して返す（Instagram 投稿に適した形式。共通骨格は Content-Type: image/jpeg・拡張子 .jpg・file_format='jpg' 固定で S3 保存・DB 登録するため、実運用方式は JPEG を返す前提。この固定は Phase 14-4 の契約一般化に伴い、14-7 で方式が返す file_format ベースの保存に変わる予定）。

  • モードが RGB でない場合（アルファチャンネル付き等）は convert("RGB") で RGB 化する
  • JPEG 品質は quality=90 固定（prompt_configs.parameters の quality は API の生成品質であり、JPEG エンコード品質とは別物）

7. エラーハンドリング

方式内に try/except・リトライは持たない。Secret 取得失敗・パラメータ不正・API 例外・画像変換失敗はすべてそのまま送出し、共通骨格（processing.py）が prompt_config 単位で捕捉してスキップする。未完了の prompt_config が残るとバッチは失敗終了し、Step Functions の Retry → 失敗アラーム通知につながる（batch-flow.html セクション 2 の完了判定・失敗時ポリシー）。再実行時は共通骨格の完了判定により生成済み分がスキップされるため、方式は冪等性を意識しなくてよい。
