セット別・生成方式別設計書（sets/ / generators/）

セット（テーマ）が増えるたびに docs/app/sets/<set_code>.html が、生成方式が本採用されるたびに docs/app/generators/<方式名>.html が 1 本ずつ増える（3 層構造: 契約 / 方式 / セット。Phase 10-1 で 2 層から拡張）。共通設計書（上記の親ページ + 3 分冊）はセット追加・方式追加では変更しない（方式カタログへの 1 行追加を除く）。 ルール・標準記載項目はアプリ設計大枠 セクション 1.1 を参照。生成方式の一覧・設計書へのリンクは方式カタログ（batch-flow.html セクション 2.1）で管理する（ここには重複掲載しない）。セットの一覧は以下。

日付論点メモ
2026-07-12 セット別生成ロジックの隔離方針（アプリ資材の共有と拡張性の両立） D-3 通読時の議論で合意。将来の多段生成 AI チェーン（プロンプト生成 AI → 画像生成 AI → 品質向上 AI 等）でセットごとに生成ロジックが複雑化する懸念に対し、アプリ資材（コンテナ・パイプライン）はサービス単位の共有を維持しつつ、image-batch 内部を「共通骨格（冪等性アンカー・S3 保存・DB 登録）+ 生成方式 strategy モジュール（generators/<方式名>.py）+ 方式名レジストリ」で構成する。方式の割当は DB のセット設定（方式名の設定値）で行い、コードに set_code の分岐は書かない。新方式の追加は「新規ファイル + レジストリ 1 行」に閉じ、既存方式のファイルには触れない。共有部品（core / API クライアント）の変更はパイプラインの pytest ゲートで守る。段階的エスカレーション（デプロイ分離 = Scheduler payload でセットごとにタスク定義リビジョンを固定 → 資材分離 = サービス切り出し）も整理したが、現時点では採用しない（デプロイ分離を導入する場合、結合点は DB スキーマの後方互換性に移る点に注意）。
→ 反映済み（2026-07-15、Phase 10-1）: batch-flow.html セクション 2.1（strategy 構造の decision）と design-outline.html セクション 1.1（採用しない選択肢の整理）に反映。
2026-07-12 生成方式の設計書による batch-flow.html の肥大化防止（3 層分冊） 生成方式はスクラップ&ビルドで多数試作する想定のため、方式の設計を batch-flow.html に直接書き足さない。batch-flow.html には契約（方式インターフェース: セット設定を受け取り最終画像を返す。冪等性・S3・DB 登録は共通骨格の責務で方式は関与しない）と方式カタログ（1 方式 1 行: 方式名・概要・ステータス〔試行中 / 稼働中 / 廃止〕・設計書リンク）のみを置き、方式本体の設計は docs/app/generators/<方式名>.html に 1 方式 1 本で分冊する。試行段階では設計書を書かず、セットが本採用した時点で作成する。廃止方式は削除せず廃止マークで履歴として残す（セット別設計書と同じ扱い）。design-outline.html セクション 1.1 の 2 層構造（共通 / セット別）に方式層を加えた 3 層（契約 / 方式 / セット）として反映する。
→ 反映済み（2026-07-15、Phase 10-1）: design-outline.html セクション 1.1 の 3 層化と、batch-flow.html セクション 2.1 の契約・方式カタログ新設で反映。
