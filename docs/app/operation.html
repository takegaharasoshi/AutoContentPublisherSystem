手順 1: Secrets Manager に SNS 認証 Secret を作成する

名前規約・値のスキーマは infra/security.html セクション 1.2〜1.3（構成要素の意味付けはセクション 5.1 で最終確認済み）。

aws secretsmanager create-secret \
  --name acps/prod/fashion-set-2/sns/instagram/main-account \
  --secret-string '{"access_token":"...","ig_user_id":"...","token_expires_at":"2026-09-05T00:00:00Z"}'

手順 2: DB にセットのレコードを登録する

Query Editor で以下の順に INSERT する（後続テーブルが set_id を参照するため、この順序を守る）。

DB 直接操作の共通注意（Phase 6 の実機確認より。本書のすべての SQL 操作に共通）
  • Query Editor の接続時は「データベースを変更する」で DB 名 acps を必ず指定する。未指定のまま実行すると No database selected (1046) で失敗する
  • Aurora が自動一時停止中に Query Editor / Data API を実行すると DatabaseResumingException が返る。アクセス自体が再開のトリガーになるため、数十秒待って再試行する
  • Query Editor の「最近」タブでは SQL 中の日本語が文字化けして表示されるが、表示上の問題のみで保存データは正しい（utf8mb4）
-- (1) batch_sets（generator_name は方式カタログ〔batch-flow.html セクション 2.1〕の方式名を設定する）
INSERT INTO batch_sets (set_code, name, description, generator_name, is_active)
VALUES ('fashion-set-2', 'ファッション系セット2', 'セット2の説明', '(方式名)', 1);
-- 発行された id を以降の set_id として使う

-- (2) prompt_configs（初期運用は is_active=1 を 1 件のみ）
INSERT INTO prompt_configs (set_id, prompt_text, negative_prompt, parameters, is_active)
VALUES (:set_id, '(プロンプト本文。1 枚だけ生成されるよう文言で制御)', NULL, NULL, 1);

-- (3) caption_templates（0 件でも投稿は継続できるが、通常は 1 件登録する）
INSERT INTO caption_templates (set_id, template_text, is_active)
VALUES (:set_id, '(キャプション本文 + ハッシュタグ)', 1);

-- (4) sns_accounts（手順 1 で作成した Secret の account_code と一致させる）
INSERT INTO sns_accounts (set_id, platform, account_code, account_name, is_active)
VALUES (:set_id, 'instagram', 'main-account', 'アカウント表示名', 1);

手順 3: CDK の ImageBatchStack に EventBridge Scheduler を追加して deploy する

入力に set_code を指定したスケジュールを 1 件追加する（1 日複数回投稿する場合は複数時刻分）。cron 式・Scheduler 定義の書き方は infra/workflow.html、変更後のデプロイは cdk deploy -c env=prod ImageBatchStack（イメージタグ 3 つの Context 指定が必須。指定ルール・現行タグの確認方法は infra/cicd.html セクション 3.2）。

初セットは既存プレースホルダ Scheduler を転用する（Phase 10-1 でインフラ実態を反映） ImageBatchStack には Phase 5 で作成した Scheduler acps-prod-image-generation-schedule（State=DISABLED・プレースホルダ cron・入力 set_code=fashion-set-1）が既に存在する（infra/workflow.html セクション 1.5 の decision）。 初セットの追加では新規 Scheduler を追加せず、このプレースホルダの cron 式を本番の投稿時刻に書き換え、State を ENABLED にして deploy する。2 セット目以降が本手順どおりの「Scheduler 1 件追加」となる。

手順 4: 動作確認

画像生成 Step Functions を手動実行し（infra/operation.html セクション 1.2 の「手動での画像生成実行」）、CloudWatch Logs で当該 set_code の生成・投稿が成功することを確認する。

セット追加チェックリスト
  • セット別設計書（docs/app/sets/<set_code>.html）を作成した
  • Secrets Manager に acps/{env}/{set_code}/sns/{platform}/{account_code} を作成した（DB 登録より先）
  • batch_sets → prompt_configs → caption_templates → sns_accounts の順に DB へ登録した
  • ImageBatchStack に Scheduler を追加して cdk deploy した（イメージタグ 3 つを Context 指定。infra/cicd.html セクション 3.2）
  • 手動実行で生成・投稿の成功を CloudWatch Logs で確認した
セット追加で CDK スタックは増えない スタック（FoundationStack / ImageBatchStack / SnsPostBatchStack / MonitoringStack）は機能単位であり、セット単位ではない。 セットを追加しても新しいスタックは作られず、増えるのは既存 ImageBatchStack 内の Scheduler 定義 1 件 + Secrets Manager の Secret + DB レコードのみ。 Step Functions・ECS タスク定義・コンテナイメージは全セット共有で、Scheduler が渡す set_code によってどのセットの処理を行うかが決まる。 したがって、インフラ設計書もセット追加では変更不要（2026-07-06 ユーザー質問を受けて明記）。

2.2 セット廃止・入れ替え（スクラップ&ビルド）

  1. batch_sets.is_active=0 に更新する（両バッチは is_active=0 のセットをスキップして正常終了する）。この時点でスケジュールが残っていても実害はないため、順序を厳密に守る必要はないが、次のステップより先に行うことを推奨する
  2. CDK から該当セットの Scheduler 定義を削除して deploy する（イメージタグ 3 つの Context 指定が必須。infra/cicd.html セクション 3.2）
  3. データ（DB レコード = メタ情報・投稿履歴）は削除せず残す。S3 実体（画像ファイル）はインフラの 30 日ライフサイクルで自動削除される対象であり「残す」対象に含まない（Phase 9-3 で確定。データモデル設計 セクション 5）
  4. セット別設計書は削除せず、廃止の旨（バッジ・廃止日）を付けて履歴として残す

3. プロンプト・キャプション管理

prompt_configs・caption_templates は DB 直接操作（Query Editor 等）で登録・変更する。管理画面は将来拡張。生成時にはプロンプトのスナップショットが generated_images に、キャプションのスナップショットが posts に記録されるため、変更しても過去の生成・投稿記録は影響を受けない。

軽微な文言修正（履歴を残さない場合）

既存の有効行を直接 UPDATE する。過去の生成実行への影響はスナップショットにより発生しない。

UPDATE prompt_configs SET prompt_text = '(新しいプロンプト本文)' WHERE id = :prompt_config_id;
UPDATE caption_templates SET template_text = '(新しいキャプション本文)' WHERE id = :caption_template_id;

差し替え（旧設定を無効化して残す場合）

新しいレコードを追加し、旧レコードを is_active=0 にする。prompt_configs は「is_active=1 が 0 件」だと画像生成バッチが異常終了する（batch-flow.html セクション 2）ため、必ず 1 トランザクションで実行し、0 件になる瞬間を作らない。

START TRANSACTION;
INSERT INTO prompt_configs (set_id, prompt_text, negative_prompt, parameters, is_active)
VALUES (:set_id, '(新しいプロンプト本文)', NULL, NULL, 1);
UPDATE prompt_configs SET is_active = 0 WHERE id = :old_prompt_config_id;
COMMIT;

caption_templates は is_active=1 が 0 件でも投稿処理は継続する（キャプションなしで投稿。data-model.html セクション 4.3）ため、トランザクションは必須ではないが同様の手順で問題ない。

セット別拡張ポイント（再掲） AI によるプロンプト生成・AI キャプション生成は将来のセット別拡張ポイント（バッチ処理フロー設計 セクション 2・3）。どのセットがどの方式かはセット別設計書に記録する。本セクションの手順は「固定文を DB で管理する」初期実装に対するものであり、AI 生成方式導入時は別途手順を定める。

4. 投稿失敗時の対応

初期実装は自動再試行なし・通知のみ（欠番許容）。1 日複数回投稿するため、1 回の欠番は致命的でないという判断（A-1 壁打ち記録 セクション 4）。

4.1 確認手順

  1. 失敗時はメール通知（MonitoringStack のアラーム / EventBridge Rule。仕組みは infra/workflow.html セクション 7〜10）を受ける
  2. Step Functions 実行履歴・CloudWatch Logs で、どのステートで・どのアカウントで失敗したかを確認する
  3. posts テーブルで失敗の詳細（status / error_message / api_response）を確認する: 
    SELECT p.id, p.status, p.error_message, p.api_response, p.updated_at,
       sa.account_code, gr.scheduled_at
FROM posts p
JOIN sns_accounts sa ON sa.id = p.sns_account_id
JOIN generation_runs gr ON gr.id = p.generation_run_id
JOIN batch_sets bs ON bs.id = gr.set_id
WHERE bs.set_code = 'fashion-set-1'
ORDER BY p.updated_at DESC
LIMIT 20;
  4. 必要な場合のみ、SNS 投稿ワークフロー（sns-posting-sfn）を手動で単独起動する（infra/operation.html セクション 1.2）。再実行は必ず Step Functions 経由で行い、ECS RunTask の直接実行は使わない（バッチは DB 接続リトライを持たず、Aurora 自動一時停止中の直接起動は失敗するため。batch-flow.html セクション 1 の前提）。投稿対象の決定ロジックは自動実行と同じため、その時点で最古の actionable な生成実行（data-model.html セクション 4.4）が処理される。一度試行して失敗した分は自動では再対象にならない（意図的に再対象にしたい場合は 4.2 を参照）

4.2 手動補正手順

DB を手動補正する前に、対象の set_code・post_id・現在の値を development-plan.md の「トラブルシューティングログ」に記録してから更新する。

published_unconfirmed を投稿済みに補正する

Instagram 側で投稿済みを確認できた場合のみ実行する。

UPDATE posts
SET status = 'success',
    platform_post_id = '(確認できた Instagram media ID)',
    posted_at = '(確認できた投稿日時 UTC)',
    error_message = NULL
WHERE id = :post_id
  AND status = 'published_unconfirmed';

failed / published_unconfirmed を未投稿として再試行可能にする

Instagram 側で未投稿を確認し、原因を解消した上で実行する。posts は UNIQUE (generation_run_id, sns_account_id) のため新規行は作れず、既存行を pending に戻して「最初から実行」扱いにする（batch-flow.html セクション 3.2 の復旧ロジック）。Instagram のメディアコンテナは時間経過で無効になり得るため、platform_container_id を含めてクリアし常にコンテナ作成からやり直す。

UPDATE posts
SET status = 'pending',
    platform_container_id = NULL,
    platform_post_id = NULL,
    caption_template_id = NULL,
    caption_text_snapshot = NULL,
    error_message = NULL,
    api_response = NULL
WHERE id = :post_id
  AND status IN ('failed', 'published_unconfirmed');
注意: 補正しても即座には処理されない場合がある SNS 投稿バッチは常にその時点で最古の actionable な生成実行 1 件のみを処理する（batch-flow.html セクション 3.1）。補正対象より古い actionable な生成実行が他に残っている場合、次回実行（自動 / 手動を問わず）はそちらが先に処理される。補正対象を優先させたい場合は、以下のクエリ（data-model.html セクション 4.4のクエリを転用）で、補正対象の generation_run_id がその時点で最古の actionable であることを確認してから手動実行する。 
SELECT gr.id
FROM generation_runs gr
WHERE gr.set_id = :set_id
  AND EXISTS (
    SELECT 1 FROM sns_accounts sa
    WHERE sa.set_id = gr.set_id AND sa.is_active = 1
      AND NOT EXISTS (
        SELECT 1 FROM posts p
        WHERE p.generation_run_id = gr.id
          AND p.sns_account_id = sa.id
          AND p.status IN ('success','failed','published_unconfirmed')
      )
  )
ORDER BY gr.scheduled_at ASC, gr.id ASC
LIMIT 1;
古い actionable な生成実行が他にもある場合は、4.3 の stale データの扱いを先に検討する。

4.3 stale データ（未完了のまま残った生成実行）の扱い

data-model.html セクション 4.4 で確認したとおり、pending / container_created のまま終端状態に達しない posts 行が残ると、その生成実行は後続の実行でも自動的に再選択され続ける。これは主に、ECS タスクの異常終了（OOM Kill・Fargate 基盤障害等）でアプリケーションが例外を捕捉して failed に更新する前に強制終了した場合に起こる。投稿対象決定は「最古の actionable」を選ぶ方式のため、この状態の生成実行が残り続けると同一セットの後続投稿がブロックされる点が運用上の注意点である。

決定: 自動検知の仕組みは追加しない（既存アラーム 1 本に一本化する方針を維持） stale 専用の CloudWatch アラームやカスタムメトリクスは追加しない（batch-flow.html セクション 3.4 の「通知は既存の Step Functions 失敗アラーム 1 本に一本化」という決定と整合させる）。運用者は既存の失敗通知を起点に、以下のクエリで stale な行の有無を確認する運用とする。 
-- 各セットの actionable な (generation_run, account) を一覧表示（古い順）
SELECT bs.set_code, gr.id AS generation_run_id, gr.scheduled_at,
       sa.account_code, p.status, p.updated_at
FROM generation_runs gr
JOIN batch_sets bs ON bs.id = gr.set_id
JOIN sns_accounts sa ON sa.set_id = gr.set_id AND sa.is_active = 1
LEFT JOIN posts p ON p.generation_run_id = gr.id AND p.sns_account_id = sa.id
WHERE bs.is_active = 1
  AND (p.id IS NULL OR p.status NOT IN ('success', 'failed', 'published_unconfirmed'))
ORDER BY gr.scheduled_at ASC;
  • 判定の目安: 1 日複数回実行する運用のため、status が pending / container_created のまま updated_at から半日以上経過している行は stale の疑いが強い（対応する Step Functions 実行がすでに終了しているにもかかわらず DB が更新されていない状態）。行が存在しない（p.id IS NULL）だけの場合は、直近の生成実行がまだ処理待ちなだけの通常状態であり stale ではない
  • 対応: 対象の Step Functions 実行履歴・CloudWatch Logs で実際の投稿有無を確認し、4.2 の手順で success（実際は投稿済みだった場合）または failed（実際は未投稿だった場合）の終端状態に補正する。終端状態に補正することで、後続の投稿対象決定が次の生成実行へ進めるようになる
  • 将来的にセット別失敗時ポリシー（追いつく / 捨てる。batch-flow.html）を導入する際は、stale データの扱いも合わせて再設計する

5. Secret 運用（SNS 認証・画像生成 API キー）・トークン更新

5.1 Secret 名規約の最終確認

Secret 名規約 acps/{env}/{set_code}/sns/{platform}/{account_code}（infra/security.html セクション 1.2）と、構成要素の意味付け（set_code = batch_sets.set_code、account_code = sns_accounts.account_code）はアプリ設計大枠 セクション 5で確定済みであり、本規約に変更はない。account_code は Secret 名導出専用のため作成後変更不可。変更が必要な場合は新アカウント（新しい account_code）として追加し、旧アカウントは is_active=0 にする。

Phase 10-1 での再確認（2026-07-15 実施済み） 本規約は Phase 0〜8（インフラ構築）を経ずに Phase 9 として前倒しで確定したものであるため、Phase 10-1（アプリ設計の最終 Fix）でインフラ実態と突き合わせて再確認した。 IAM ポリシー（プレフィックスベースの Secret 読み取り権限。infra/security.html セクション 2.1）・Secret 運用の実態と矛盾はなく、規約は変更なしで最終確認とする。

5.2 既存セットへの SNS アカウント追加手順

新規セット追加（セクション 2.1）とは異なり、既存セットに投稿先アカウントを追加する場合（将来の多プラットフォーム展開等）の手順。

  1. Secrets Manager にシークレットを作成する（DB 登録より先に行う）。値のスキーマは infra/security.html セクション 1.3 を参照 
    aws secretsmanager create-secret \
  --name acps/prod/fashion-set-1/sns/instagram/sub-account \
  --secret-string '{"access_token":"...","ig_user_id":"...","token_expires_at":"2026-09-05T00:00:00Z"}'
  2. sns_accounts にレコードを追加する。account_code はシークレット名と一致させ、作成後は変更しない 
    INSERT INTO sns_accounts (set_id, platform, account_code, account_name, is_active)
VALUES (:set_id, 'instagram', 'sub-account', 'サブアカウント表示名', 1);
  3. 動作確認: SNS 投稿ワークフローを手動実行し、追加したアカウントへの投稿（またはシークレット取得）が成功することを確認する

5.3 Instagram アクセストークン更新の具体手順とリマインド運用

  1. Instagram Graph API の長期アクセストークンは発行から60 日で失効する。失効前（目安: 失効の 7 日前まで）に Meta の長期トークン更新エンドポイントを呼び出し、新しいトークンと新しい失効日時を取得する
  2. 取得した新トークンで Secrets Manager の値を更新する（access_token と token_expires_at の両方を更新する） 
    aws secretsmanager put-secret-value \
  --secret-id acps/prod/fashion-set-1/sns/instagram/main-account \
  --secret-string '{"access_token":"(新トークン)","ig_user_id":"...","token_expires_at":"2026-11-04T00:00:00Z"}'
  3. 更新後、SNS 投稿ワークフローを手動実行するか次回の自動実行を待って、新トークンで認証が成功することを確認する
決定: リマインド運用はアカウントごとの手動管理とする（自動リマインドは導入しない） token_expires_at は Secret 内の運用メモであり、アプリケーションはこの値を認証には使用しない（infra/security.html セクション 1.3）。管理画面（将来拡張）がない現時点では、Secrets Manager から失効日時を横断的に一覧する仕組みも持たない。 そのため初期運用では、アカウント作成・トークン更新のたびに、失効日の 7 日前を目安に運用者自身がカレンダー等の外部ツールにリマインダーを設定する運用とする（セット数が少ない初期スコープでは許容範囲と判断）。 セット数が増えて手運用が破綻する場合は、Secrets Manager を横断的に読み取り token_expires_at が近いものを通知する仕組み（Lambda + EventBridge 等）の導入を将来検討する（設計課題リストに記録）。

5.4 画像生成 API キーの設定（Phase 10 開始時に必須）

画像生成 API キー用 Secret acps/prod/image/api-key は、インフラ構築（Phase 5-3）で「箱」のみ作成済みで、値はランダムなプレースホルダのままである（infra/stacks.html セクション 3.1 の decision）。アプリ実装（Phase 10）で画像生成 API を呼び出す前に、実際の API キーへ差し替える。

  1. マネジメントコンソールの Secrets Manager で acps/prod/image/api-key を開き、「シークレットの値を取得」→「編集」で api_key キーの値を、利用する画像生成 API（初期方式が使う API。方式カタログは batch-flow.html セクション 2.1）の API キーに差し替える。値スキーマ {"api_key": "..."} は infra/security.html セクション 1.3 を参照
  2. 差し替え後、画像生成ワークフローの実行で認証が成功することを確認する
注意: スタック再作成で値が失われる この Secret の RemovalPolicy は Delete のため、FoundationStack を destroy → 再作成すると手動設定した API キー値は失われる。再作成後はコンソールから再設定が必要（infra/stacks.html セクション 3.1）。

6. Phase 9-4 の決定事項まとめ

  • セット追加手順を 5 ステップ（設計書雛形 → Secret 作成 → DB 登録（batch_sets → prompt_configs → caption_templates → sns_accounts の順）→ Scheduler 追加 deploy → 動作確認）に具体化し、チェックリスト化した（2.1）
  • プロンプト・キャプションの変更は「直接 UPDATE（履歴を残さない）」と「新規 INSERT + 旧行 is_active=0（履歴を残す）」の 2 方式を定義し、prompt_configs は 0 件化を避けるためトランザクションで実行することを確定した（3）
  • 投稿失敗時の手動補正は、旧設計の attempt_number ベースの再試行を採用せず、posts 行を pending に戻して「最初から実行」扱いにする方式で再投稿可能にすると確定した。あわせて、補正しても投稿対象決定が「最古の actionable」を優先するため即座に処理されるとは限らない点を明記した（4.2）
  • stale データ（pending / container_created のまま残った生成実行が後続投稿をブロックする問題）について、専用の自動検知は追加せず、既存の失敗アラームを起点に定期確認クエリで検出し 4.2 の手順で終端状態に補正する運用と確定した（4.3）
  • SNS Secret 規約 acps/{env}/{set_code}/sns/{platform}/{account_code} は変更なしで最終確認とし、既存セットへのアカウント追加手順（Secret 作成 → DB 登録の順）を新規セット追加手順と分けて明文化した（5.1〜5.2）
  • Instagram トークン更新のリマインドは、管理画面がない初期スコープでは運用者による手動リマインダー設定とし、自動化は将来課題として設計課題リストに記録した（5.3）
