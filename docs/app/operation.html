アプリ運用設計（骨子）

Phase A-4 骨子版（Phase 9-4 で詳細化） 最終更新: 2026-07-06
本書の位置づけ セット追加・廃止、プロンプト / キャプション管理、投稿失敗時対応などアプリ（業務）の運用を定める設計書。Phase A-4 時点は骨子版であり、 具体手順は Phase 9-4 で詳細化する（セクション 6）。 インフラの運用（デプロイ、Aurora 一時停止、監視・通知の仕組み、コスト）は infra/operation.html を参照。 全体方針・分冊間の整合は親ページ アプリ設計大枠 を参照。

1. 運用方針一覧（初期実装）

運用方針（初期実装）
セット追加 セクション 2.1 の 4 ステップ（セット別設計書の作成 → Secret 作成 → DB 登録 → Scheduler 追加 deploy）

2. セット追加・廃止

2.1 セット追加手順（骨子）

  1. セット別設計書を作成する — docs/app/sets/<set_code>.html に、テーマ・アカウント構成・スケジュール・プロンプト / キャプションの方式選択・失敗時ポリシー等を記録する（ルールと標準記載項目はアプリ設計大枠 セクション 1.1）
  2. Secrets Manager に SNS 認証 Secret を作成する — 名前規約 acps/{env}/{set_code}/sns/{platform}/{account_code}（infra/security.html）
  3. DB にセットのレコードを登録する — batch_sets / prompt_configs / caption_templates / sns_accounts（Query Editor 等で直接操作）
  4. CDK の ImageBatchStack に EventBridge Scheduler を追加して deploy する — 入力に set_code を指定したスケジュールを 1 件追加する（1 日複数回投稿する場合は複数時刻分）
セット追加で CDK スタックは増えない スタック（FoundationStack / ImageBatchStack / SnsPostBatchStack / MonitoringStack）は機能単位であり、セット単位ではない。 セットを追加しても新しいスタックは作られず、増えるのは既存 ImageBatchStack 内の Scheduler 定義 1 件 + Secrets Manager の Secret + DB レコードのみ。 Step Functions・ECS タスク定義・コンテナイメージは全セット共有で、Scheduler が渡す set_code によってどのセットの処理を行うかが決まる。 したがって、インフラ設計書もセット追加では変更不要（2026-07-06 ユーザー質問を受けて明記）。

2.2 セット廃止・入れ替え（スクラップ&ビルド）

  • batch_sets.is_active=0 に更新する（両バッチは is_active=0 のセットをスキップして正常終了する）
  • CDK から該当セットの Scheduler 定義を削除して deploy する
  • データ（DB レコード = メタ情報・投稿履歴）は削除せず残す。S3 実体（画像ファイル）はインフラの 30 日ライフサイクルで自動削除される対象であり「残す」対象に含まない（Phase 9-3 で確定。データモデル設計 セクション 5）
  • セット別設計書は削除せず、廃止の旨（バッジ・廃止日）を付けて履歴として残す

3. プロンプト・キャプション管理

  • プロンプト（prompt_configs）・キャプションテンプレート（caption_templates）は DB 直接操作（Query Editor 等）で登録・変更する。管理画面は将来拡張
  • 生成時にはプロンプトのスナップショットが generated_images に記録されるため、変更しても過去の生成記録は影響を受けない
  • プロンプト / キャプションを AI で生成する方式は将来のセット別拡張ポイント（バッチ処理フロー設計 セクション 2・3）。どのセットがどの方式かはセット別設計書に記録する

4. 投稿失敗時の対応

初期実装は自動再試行なし・通知のみ（欠番許容）。1 日複数回投稿するため、1 回の欠番は致命的でないという判断（A-1 壁打ち記録 セクション 4）。

  1. 失敗時はメール通知（MonitoringStack のアラーム / EventBridge Rule。仕組みは infra/workflow.html セクション 7〜10）を受ける
  2. Step Functions 実行履歴・CloudWatch Logs・DB（posts / generation_runs）で失敗箇所と原因を確認する
  3. 必要な場合のみ、SNS 投稿ワークフロー（sns-posting-sfn）を手動で単独起動する — 投稿対象の決定ロジックは自動実行と同じため、未試行の生成実行が投稿される（バッチ処理フロー設計 セクション 3）。一度試行して失敗した分は自動では再対象にならない（手動補正の具体手順は Phase 9-4）

5. SNS Secret 規約の運用・トークン更新

  • Secret 名規約 acps/{env}/{set_code}/sns/{platform}/{account_code} の定義は infra/security.html、構成要素の意味付けはアプリ設計大枠 セクション 5 で確定済み
  • account_code（sns_accounts.account_code）は作成後変更不可（Secret 名導出専用）。変更が必要な場合は新アカウントとして追加する
  • Instagram のアクセストークンは 60 日で失効するため、失効前に手動でリフレッシュし Secret を更新する（旧設計の運用を踏襲。手順詳細・リマインド方法は Phase 9-4）
