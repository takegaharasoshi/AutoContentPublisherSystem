本書の位置づけ Phase 14-2「コンテンツ生成（動画）対応の要件壁打ち」（2026-07-23 実施）の記録であり、設計書ではない。 Phase 14-3 以降（設計大枠 → 詳細設計 → 実装）の入力として使用する。 壁打ち記録は 1 回 1 本のパターン（14-1 で決定）に基づく新規ノート。A-1 の壁打ち記録は requirements-notes.html を参照。 上位の事業判断（収益化経路・予算・KPI）は事業戦略書を正とする。

1. 背景と目的

  • 動画（リール）対応は収益化経路（Meta ボーナス = リール再生数ベース）の必要条件であり、Phase 14 の最重要機能（事業戦略書 セクション 2）。KPI マイルストーン M1「リール自動投稿の定常運用」に対応する
  • 初期スコープは「静止画 + 音楽」の簡易動画方式（ffmpeg 等のローカル合成。生成 API 費ほぼゼロ）から開始し、生成 AI 動画方式は generator strategy の差し替えで後から導入する — 14-1 の壁打ちで合意済み。本壁打ちはこの方式の要件を具体化する
  • 初期パラメータ感「1 日 1 本・10 秒前後・標準画質」（事業戦略書 セクション 3.1）を出発点とする

2. 調査結果（決定の前提となる事実。2026-07-23 時点）

2.1 API 経由のリール投稿と音源の制約

確定事実: API 投稿では Instagram の音楽ライブラリは使えない
  • Instagram の音楽ライブラリ・トレンド音源の付与はネイティブアプリ限定機能であり、Graph API 経由のリール投稿では利用できない。音声は動画ファイルに埋め込んで渡すしかない
  • そもそも Instagram の音楽ライブラリは個人・非商用利用向けライセンスであり、ビジネスアカウント・商用利用には権利が付与されない
  • 権利処理されていない音源を埋め込むと、公開後に自動ミュート・削除の対象となり得る
したがって自前で権利クリアした音源を調達し、合成時に埋め込む方式が必須となる（開発計画の事前調査どおり）。

2.2 自前音源の調達選択肢の比較

選択肢費用商用利用プラットフォーム制限備考
CC0・フリーライセンス音源
（Pixabay License 等）		 無料 可 なし（TikTok 転用可） CC0 は著作権放棄（クレジット不要）。CC-BY は要クレジット、CC-BY-NC（非商用）は収益化コンテンツに使用不可のため除外。難点は他アカウントとの音源被りと、世界観との合致が選曲頼みになる点
Meta Sound Collection
（Meta 公式の無料音源ライブラリ）		 無料 可 Facebook / Instagram 限定 ライセンス規約上、Meta 製品外での利用・配布は不可。TikTok 展開（Phase 17 第一候補）で音源差し替えが必要になるため不採用
AI 音楽生成
（Suno 等）		 有料プラン $10/月〜 有料プラン中の生成曲のみ可
（無料プラン生成曲は不可）		 なし 契約中に生成した曲の商用ライセンスは解約後も持続するため、「1 か月だけ契約してストック生成」の単発費用パターンが可能（固定費を積まない予算方針と整合）。独自世界観に合う音源を作れ、非オリジナルコンテンツ対策にも寄与

2.3 リール投稿の技術仕様の概略

  • 推奨フォーマット: 9:16 縦長（1080x1920）・MP4（H.264 + AAC）。現行の生成画像は 1024x1024 正方形のためギャップがある
  • Graph API のリール投稿はコンテナ作成時に media_type=REELS と video_url（公開アクセス可能な URL）を渡す方式。現行の S3 presigned URL 方式（batch-flow.html セクション 3.3）は動画でも流用できる見込み
  • 動画はサーバ側の処理（コンテナが FINISHED になるまで）が画像より長く、分オーダーになり得る → ポーリング設計・タイムアウトの見直しが必要（セクション 6）
  • share_to_feed パラメータでプロフィールのフィード面（グリッド）への表示可否を制御できる

3. 決定事項（壁打ち結果）

#論点決定理由
1 音源の調達方針 段階方式。検証用セット（fantasy-animals-1）は CC0・フリーライセンス音源で開始し、勝負セット投入（Phase 15）時に AI 音楽生成（Suno 等）への格上げを検討する 無料・固定費ゼロで開始でき、プラットフォーム制限がなく TikTok 転用も可。AI 生成は単発費用パターン（2.2 参照）で導入でき、独自性が要る勝負セットで効果が大きい。Meta Sound Collection は Meta 限定ライセンスのため不採用
2 動画の画角 縦長画像（1024x1536、2:3）を生成し、ffmpeg 合成時に 9:16 へ微調整（軽いクロップまたはパディング） リールで全画面表示され見栄えが最良。変更は prompt_configs の params（size）のみで生成コストもほぼ同等
3 動画演出（簡易動画方式の中身） 静止画 1 枚 + ゆっくりズーム/パン（Ken Burns 効果）+ 音楽。長さ 10 秒前後・標準画質 ffmpeg のみで実装でき、「1 起動 = 1 枚生成」の現行不変条件を維持できる。完全静止よりリールらしい動きが出る。複数枚スライドショーは生成費・改修規模が増えるため将来拡張
4 fantasy-animals-1 の実行配分 全て動画（リール）に置換し、実行回数を 1 日 3 回 → 1 日 1 回に削減（時刻は実装時に決定） セットの generator_name を動画方式へ差し替えるだけで構造変更が最小。1 セット内で静止画・動画を併用する場合は方式割当をセット単位からスロット/設定単位へ拡張する必要があり、検証用セットには過剰。回数削減で API 費約 1/3 とし勝負セットへ予算温存（初期パラメータ感「1 日 1 本」とも一致）
5 静止画方式の扱い gpt-image-single 方式は廃止しない。コード（strategy モジュール）・方式設計書・方式カタログ掲載はそのまま残す ユーザー要件「将来的に画像だけの投稿も行う可能性があるため、生成方式は消さない」。方式はセットへの割当を差し替えるだけであり、割当を戻す（または画像方式の別セットを作る）ことでいつでも再利用できる
6 音源の運用 セットごとに複数曲（3〜5 曲目安）を S3 にストックし、投稿ごとにローテーション。全曲について出典 URL・ライセンス種別・取得日の証跡を残す 毎回同じ曲による飽き・量産感を軽減（非オリジナルコンテンツ対策の観点でも有利）。prompt_configs / caption_templates と同じ「セット配下の有効レコード」パターンで設計できる
7 AI 利用の開示（事業戦略書セクション 2 の必須要件） ハッシュタグで開示する（#AIart 等。具体タグは設計時に選定）。キャプションテンプレートに含める運用ルールとして義務化 テーブル構造の変更が不要で、テンプレート経由により全投稿へ確実に付与される。開示文言の追加は必要になれば運用（テンプレート更新）で対応できる
8 リールのフィード面表示 表示する（share_to_feed=true） リール面とフィード/グリッドの両方に露出しリーチ最大化。フォロワー資産を KPI に置く方針と整合

4. 初期スコープ（簡易動画方式の要件まとめ）

  • 生成: 縦長静止画 1024x1536 を 1 枚生成（1 起動 = 1 枚生成 = 1 投稿の不変条件は維持）→ ffmpeg で 9:16・10 秒前後・ズーム/パン・音源合成・標準画質の MP4（H.264 + AAC）を作成
  • 音源: CC0・フリーライセンス音源をセットごとに 3〜5 曲ストックしローテーション。ライセンス証跡（出典 URL・ライセンス種別・取得日）を保持
  • 投稿: リール（media_type=REELS）として投稿。share_to_feed=true。キャプションは既存テンプレート方式 + AI 開示ハッシュタグ
  • 実行: fantasy-animals-1 を 1 日 1 回・全て動画に置換（Scheduler 3 件 → 1 件。時刻は実装時に決定）

4.1 将来拡張（今回のスコープ外）

  • 生成 AI 動画方式（Sora 等）: generator strategy の追加で導入。特定サービスに固定しない（事業戦略書 セクション 3.1）
  • AI 音楽生成音源への格上げ: 勝負セット投入（Phase 15）時に判断
  • 複数枚スライドショー演出: 1 起動複数枚生成が必要になった時点で「1 起動 = 1 枚」運用ルールとあわせて見直す
  • 静止画投稿の再開: 方式割当を戻すか、画像方式の別セットを作ることで対応（方式は残置。決定事項 #5）

5. データモデルへの影響整理

詳細設計（テーブル定義の確定）は 14-3 以降で行う。ここでは影響箇所の洗い出しと方向性のみ記録する。

対象影響内容
batch_sets 変更なし 新方式名を generator_name に設定するだけ（レコード運用）。方式割当はセット単位のままでよい（決定事項 #4 により 1 セット内の方式混在は不要）
prompt_configs 変更なし params の size を 1024x1536 に変えるのはレコード運用（スナップショット方式で追跡可能）
generation_runs 変更なし 「1 起動 = 1 投稿」の単位・冪等性キー（set_id × scheduled_at）は動画でも不変
generated_images 要検討 中間生成物（静止画）は従来どおり記録する前提。最終成果物が動画になるため、動画ファイルのメタ情報の置き場が論点。候補: (a) generated_images を「生成メディア」に一般化（file_format カラムを活用）、(b) 動画専用テーブルの新設、(c) 1 実行 1 動画の前提で generation_runs に直接持たせる — 14-3 以降で決定
S3 キー規約 拡張必要 現行規約（data-model.html セクション 5）は images/ プレフィックス・拡張子 .jpg 固定。動画（.mp4）と音源ストックの置き場の規約追加が必要
posts 要検討 投稿のメディア種別（フィード画像 / リール）の表現方法（カラムで持つか、セット・方式から導出するか）。状態遷移 5 状態（pending / container_created / success / failed / published_unconfirmed）と投稿対象決定クエリは流用できる見込み
post_images 要検討 「投稿に載せるメディア」の参照が画像前提。動画参照への対応方法（テーブルの一般化 or 動画メタ情報の設計とセットで決定）
音源管理 新設 セット配下の音源ストックを表すテーブル（仮称 audio_assets）の新設案。S3 キー・有効フラグ・ライセンス証跡（出典 URL・ライセンス種別・取得日）・ローテーション用の情報を持つ。prompt_configs / caption_templates と同じ「セット配下の有効レコード」パターン + セット境界の複合 FK パターンを踏襲する想定
caption_templates 変更なし AI 開示ハッシュタグはテンプレート本文に含める運用ルール（operation.html に運用ルールの追記は必要）

6. インフラ・実装への影響メモ（参考）

設計は 14-3 以降で行う。壁打ちで見えた影響範囲のみ記録する。

  • image-batch: コンテナへの ffmpeg 追加、動画方式の strategy モジュール新設（方式カタログへ 1 行追加）、Fargate タスクの CPU / メモリ / 一時ストレージの見直し。方式契約（「セット設定を受け取り最終画像を返す」batch-flow.html セクション 2.1）が「最終メディアを返す」への一般化を要する可能性
  • sns-post-batch: instagram_api.py のリール対応（media_type=REELS / video_url / share_to_feed）。動画のコンテナ処理は分オーダーになり得るため、ポーリング回数・間隔と Step Functions タイムアウトの見直し。presigned URL 方式は流用可
  • Scheduler / 監視: Scheduler 3 件 → 1 件への削減（実装時）。実行時間が延びる場合のアラーム閾値・タイムアウト見直しの要否確認

7. 持ち越し論点（14-3 以降で決める）

  • 動画メタ情報のテーブル設計の確定（セクション 5 の候補 a / b / c）と post_images の扱い
  • 音源テーブルの具体設計と選曲アルゴリズム（順繰り / ランダム）
  • 音源ファイルの前処理（10 秒前後への切り出し・フェード・音量正規化）を事前手動とするかバッチ内処理とするか
  • ffmpeg 合成パラメータ（出力解像度 1080x1920・フレームレート・ビットレート =「標準画質」の定義、ズームの方向・速度・フェード）
  • 方式名（レジストリキー）の命名と方式契約の一般化の要否
  • AI 開示ハッシュタグの具体選定とキャプションテンプレートの更新
  • 1 日 1 回の実行時刻（現行 7:00 / 12:00 / 21:00 JST のどれを残すか等）
  • リールのカバー画像（サムネイル）の扱い（デフォルトのままか、thumb_offset 等を指定するか）
  • フリー音源でも著作権検知の誤検知（自動ミュート）があり得る点の運用監視（投稿後のミュート検知は当面手動確認）

  • 音源テーブル: audio_assets 新設。選曲は順繰り（last_used_at 最古・未使用優先）。証跡は DB カラムで一元管理 → data-model.html セクション 4.8
  • S3 キー規約: videos/（動画 + 中間静止画）・audio/（音源ストック）を追加。ライフサイクルのプレフィックス限定というインフラ側変更が必要と判明 → data-model.html セクション 5

  • 方式名: gpt-image-kenburns。方式契約は「最終メディアを返す」に一般化 → batch-flow.html セクション 2.1

  • 1 日 1 回の実行時刻: 14-11 で決定（持ち越し）
  • カバー画像: 指定せずデフォルト（先頭フレーム）で開始し、14-8 で実機確認 → batch-flow.html セクション 3.3
  • ミュート誤検知の運用監視: 週次手動確認。検知時は音源を無効化・差し替え（投稿は削除しない）→ operation.html セクション 4.4
