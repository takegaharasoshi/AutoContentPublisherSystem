設計書体系ガイド

現行版 最終更新: 2026-07-06 本ドキュメントは設計書体系のエントリポイント
閲覧方法 設計書は HTML で記述している。GitHub 上ではソース表示になるため、ローカルでブラウザから開くか、VS Code の拡張機能（Live Preview 等）で閲覧すること。開発計画（development-plan.md）のみ、進捗チェックリストとして頻繁に編集するため Markdown を維持している。

1. 設計原則

  1. インフラとアプリの関心の分離: インフラ設計（docs/infra/）とアプリ設計（docs/app/）を明確に分ける。インフラを先に構築・Fix する。アプリ設計は 2 段階で行い、大枠（設計書構成・主要方針の骨子）は Phase A で、詳細はインフラ完成後の Phase 9 で作成する
  2. 完璧より前進（一時 Fix 運用）: 設計書は「次フェーズの作業に着手できる」水準で一時 Fix とし、先に進む。詳細はセクション 4 の Fix 基準を参照
  3. 実装を正とする: 設計と異なる実装をした場合は、実装に合わせて設計書を修正する（変更理由は備考に残す）
  4. 同じ情報を 2 箇所に書かない: 詳細の「正」は 1 つのドキュメントに置き、他からは参照リンクで辿る

3. スコープ境界: インフラ設計とアプリ設計

設計肥大化の再発を防ぐため、インフラ設計書にはアプリ仕様を書かない。境界は以下の通り。

分類インフラ設計（docs/infra/）に書くことアプリ設計（docs/app/）に書くこと ※大枠は Phase A、詳細は Phase 9
実行基盤 VPC / ECS / Aurora / S3 / Step Functions / Scheduler の構成と CDK 定義 —
ワークフロー ASL 定義、タイムアウト、Retry/Catch、環境変数の受け渡し契約（キー名のみ） 環境変数の値の意味付け・業務上の使い方
バッチ処理 DB 準備確認タスク（db-readiness-check）の仕様（インフラの一部） 画像生成・SNS 投稿の処理フロー、冪等性・再試行設計、投稿ステータス遷移
データ Aurora / S3 というデータストアの構成・設定値 テーブル定義・ER 図・DDL、S3 キー設計の詳細
セキュリティ Secret 名規約、IAM ロール、ネットワークセキュリティ SNS 認証情報の運用（アカウント追加手順・トークン更新）
運用 デプロイ手順、Aurora 一時停止、監視・通知、コスト プロンプト管理、投稿失敗時の手動補正、DDL マイグレーション
スコープガード 設計作業やレビューの中でアプリ仕様の論点（テーブル設計、投稿の再試行ルール等）が出てきた場合、インフラ設計書には書かず、docs/app/index.html の検討メモに記録してアプリ設計（大枠は Phase A、詳細は Phase 9）で扱う。

4. 設計 Fix 基準とレビュー運用ルール

「生成 AI レビューの指摘が尽きず設計が Fix しない」問題の再発防止ルール。設計書の作成・レビュー時は必ずこのルールに従う。

4.1 Fix 基準（Definition of Done）

  • 設計書は「次フェーズの作業に着手できる」水準で一時 Fix とする
  • インフラ設計の Fix 基準: 開発計画 Phase 0〜8 の各ステップが着手できる程度に、CDK リソース・設定値・手順が決まっていること
  • 「生成 AI の指摘ゼロ」を Fix 条件にしない。AI は何度でも改善提案を出せるため、指摘ゼロを目指すと原理的に終わらない
  • 実装中に判明した設計の誤りは、実装に合わせて設計書を修正すればよい（設計原則 3）。だからこそ一時 Fix で先に進んでよい

4.2 レビュー運用ルール

  1. 観点を限定する: 生成 AI にレビューさせる際は、必ず観点を明示指定する（レビュープロンプトのテンプレートは development-plan.md の Phase D 備考を参照）。観点を指定しないフリーレビューは行わない
  2. 指摘を分類する: 指摘は「blocker（誤り・矛盾・欠落で、次フェーズの作業を妨げるもの）」と「改善提案（あった方が良いが、なくても作業は進められるもの）」に分類する
  3. blocker のみ修正する: 改善提案は修正せず、development-plan.md の「設計課題リスト」に記録して Fix を妨げない
  4. レビューは最大 2 巡まで: 3 巡目以降の指摘はすべて設計課題リスト行きとし、レビューを打ち切って Fix を宣言する
  5. YAGNI: 将来拡張（AdminApiStack 等）は「拡張ポイントがある」ことの記載まで。詳細設計はしない

5. 更新ルール

  • コードやインフラに変更を加えた際は、対応する設計書も同時に更新する
  • 設計と異なる実装をした場合は、実装に合わせて設計書を修正し、変更理由を備考（decision コールアウト）に残す
  • 同じ情報を複数ドキュメントに書かない。詳細は 1 箇所に書き、他からは参照リンクを張る
  • 設計書は HTML（本体系）で記述する。開発計画のみ Markdown

6. 旧設計書（_archive）について

docs/_archive/ には旧 Markdown 設計書を保管している。これらは現役の設計書ではない。特にアプリ設計（バッチ処理フロー・DB スキーマ・冪等性設計）はよく検討された内容を含むため、Phase 9 のアプリ設計で参考資料として活用する。
