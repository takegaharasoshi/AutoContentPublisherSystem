ワークフロー実装仕様

一時 Fix 最終更新: 2026-07-06

Step Functions・EventBridge Scheduler・DB 準備確認タスク・監視リソースの実装仕様を定義する。アーキテクチャ上の位置付けは architecture.html、CDK リソース定義は stacks.html、運用時の対応手順は operation.html を参照。

アプリ仕様は未確定 本ドキュメントの ASL・環境変数はワークフローの「器」の定義であり、業務ロジックに依存しない。ECS タスク内部の処理内容（画像生成・SNS 投稿のロジック）はアプリ設計（docs/app/）で定義する。

1. ワークフロー構成

1.1 順次実行

画像生成バッチと SNS 投稿バッチは順次実行される（全体像は architecture.html セクション 1 の図を参照）。

  1. EventBridge Scheduler が cron 式で image-generation-sfn を起動する
  2. image-generation-sfn: WaitForDbReady（DB 準備確認）→ RunImageBatchTask（画像生成バッチ）→ 成功時に StartSnsPostBatch で sns-posting-sfn を非同期起動する
  3. sns-posting-sfn: WaitForDbReady（DB 準備確認）→ RunSnsPostBatchTask（SNS 投稿バッチ）
  • SNS 投稿 Step Functions（sns-posting-sfn）は独立したステートマシンのため、手動での単独実行も可能。起動時に何が投稿されるか（投稿対象の決定）はアプリ設計（バッチ処理フロー設計 セクション 3）で定義する
  • EventBridge Scheduler は画像生成 Step Functions のみに設定し、SNS 投稿用のスケジューラは不要

1.2 共通の実装方針

  • TaskDefinition の指定方式: ASL 内の ${...TaskDefFamily} プレースホルダには、CDK が ECS Task Definition の family 名（例: acps-prod-image-batch）を渡す。ECS RunTask API は family 名指定で常に ACTIVE な最新リビジョンを使用するため、デプロイ時に ASL の更新は不要
  • SecurityGroup の指定方式: ${BatchSgId} は FoundationStack で定義される「ECS Fargate 用（バッチ共通）」Security Group を指す。画像生成バッチと SNS 投稿バッチは同一の Security Group を共有する。DB 準備確認タスクには専用の ${DbReadinessCheckSgId} を使用する
  • Task ステートのタイムアウト: ハング時にワークフローが長時間ぶら下がらないよう、WaitForDbReady=900 秒、RunImageBatchTask=3600 秒、StartSnsPostBatch=60 秒、RunSnsPostBatchTask=3600 秒の明示タイムアウトを設定する。WaitForDbReady の 900 秒は、ECS Fargate 起動（約 60 秒）+ DB 接続リトライ（最大約 510 秒）+ バッファ（約 330 秒）を考慮した値である
  • StartSnsPostBatch の呼び出し方式: arn:aws:states:::states:startExecution（.sync サフィックスなし）は非同期呼び出しであり、子実行（sns-posting-sfn）の完了を待たずに即座に成功を返す。画像生成ワークフローは SNS 投稿の起動のみを行い、投稿処理の完了は待機しない
  • StartSnsPostBatch の冪等性: StartExecution.Name には親ワークフローの $$.Execution.Name を渡す。Step Functions Standard の StartExecution は同一 Name / 同一 Input の実行が実行中であれば冪等に同じ応答を返す。一方、同名実行が既に終了済みの場合は ExecutionAlreadyExists となるため、このエラーは「子実行は既に作成済み」とみなし成功扱いにする
  • 実行名の文字数制限: Step Functions の実行名（Name）は最大 80 文字に制限されている。本システムでは $$.Execution.Name を子実行の Name として使用する。EventBridge Scheduler がデフォルトで生成する実行名は UUID ベース（36 文字）のため通常は問題ないが、カスタム名を使用する場合は 80 文字制限に注意すること
  • ResultPath の設定: WaitForDbReady と RunImageBatchTask には "ResultPath": null を設定し、ECS RunTask.sync の出力（DescribeTasks レスポンス）で入力パラメータ（$.set_code 等）が上書きされるのを防ぐ

1.3 EventBridge Scheduler の Input テンプレート

Scheduler は以下の形式で画像生成 Step Functions を起動する。<aws.scheduler.scheduled-time> は EventBridge Scheduler が実行時刻（UTC、ISO 8601）に自動置換する。

{
  "set_code": "fantasy-animals-1",
  "scheduled_at": "<aws.scheduler.scheduled-time>"
}

1.4 EventBridge Scheduler の Retry / DLQ

Scheduler には RetryPolicy と SQS DLQ を設定する。RetryPolicy は MaximumRetryAttempts=3、MaximumEventAgeInSeconds=3600 を初期値とする。リトライを使い切っても画像生成 Step Functions を起動できない場合は、失敗イベントを Scheduler DLQ（セクション 10）に送信し、MonitoringStack の CloudWatch Alarm で検知する。

2. DB 準備確認タスク（db-readiness-check）仕様

Aurora Serverless v2 の自動一時停止からの再開に対応するため、各 Step Functions ワークフローの最初のステートとして DB 準備確認 ECS タスクを実行する。バッチアプリケーション（後続の ECS タスク）は DB が利用可能な状態を前提とする。

  • ECS Task Definition: FoundationStack で定義（db-readiness-check）。両ワークフローで共有する
  • リソース: 最小構成（0.25 vCPU / 0.5 GB）
  • 配置: Public Subnet（assignPublicIp=ENABLED）。Aurora（Isolated Subnet）へ VPC 内部ルーティングで接続する
  • リトライ設計: 
    最大リトライ回数: 8
リトライ間隔: 指数バックオフ（2, 4, 8, 16, 32, 64, 128, 256 秒）
対象例外: OperationalError, InterfaceError
    Aurora Serverless v2 のコールドスタート（自動一時停止からの再開）は通常 15〜30 秒程度だが、リージョン状況や負荷によっては数分かかる場合があるため、安全側に倒して最大約 510 秒の待機時間を確保する
  • Step Functions との責務分担: WaitForDbReady ステートには Step Functions の Retry を設定しない。待機・再試行は db-readiness-check コンテナ内部で完結させ、総待機時間を上記の約 510 秒に固定する
  • 認証情報: Secrets Manager（DB 認証情報）から取得
  • 成功時: 終了コード 0 で終了し、Step Functions が次のステートへ遷移する
  • 失敗時: 終了コード 1 で終了し、Step Functions の Catch でハンドリングする
実測メモ（Phase 6-5: 一時停止からの再開挙動） 一時停止状態（0 ACU が約 90 分継続）からの手動 RunTask 検証（2026-07-14、Phase 6-5）では、初回の接続試行が Aurora の再開完了までブロックされ約 12 秒で成功し、コンテナ内リトライは発動しなかった（Phase 4-6 の SFN 経由実行でも同様に試行 1/9 で成功）。一時停止からの再開は通常、初回試行の接続待ちで吸収される。指数バックオフのリトライは、接続がエラーで即時失敗するケース（再開の長期化や一時的な障害など）に対する保険として機能する。
設計変更理由 DB 接続リトライをバッチアプリケーションコードから分離し、Step Functions ワークフローの責務とした。これにより、バッチアプリケーションは DB が利用可能な状態を前提として実装できる。
ECS Fargate 採用理由 VPC 内の Lambda はパブリック IP が付与されないため、NAT Gateway または VPC Endpoint（Interface 型）なしでは Secrets Manager にアクセスできない。ECS Fargate は assignPublicIp=ENABLED でパブリック IP を取得可能なため、追加のネットワークリソースなしで Secrets Manager にアクセスできる。起動レイテンシ（30〜60 秒）の増加はバッチ用途では許容範囲内であり、月額コストも $0.01 未満と無視できる水準。

                "Name": "StateMachineName",
                "Value": "${ImageGenerationSfnName}"
              }
            ],
            "Value": 1,
            "Unit": "Count"
          }
        ]
      },
      "Comment": "SNS 投稿 SFN の起動失敗を CloudWatch カスタムメトリクスとして発行し、MonitoringStack のアラームで検知する",
      "Next": "ImageBatchSucceeded"
    },
    "ImageBatchSucceeded": {
      "Type": "Succeed",
      "Comment": "画像生成は成功済みのため、SNS 投稿 SFN の起動失敗時もワークフローは成功終了とする。SNS 投稿が起動されなかったことはカスタムメトリクスのアラームで検知し、手動で sns-posting-sfn を起動する"
    },
    "HandleError": {
      "Type": "Fail",
      "Error": "ImageBatchFailed",
      "Cause": "Image batch task failed after retries"
    }
  }
}
StartSnsPostBatch のエラーハンドリング ExecutionAlreadyExists は子実行作成済みとして成功扱いにする。それ以外のエラーで Retry 後も起動に失敗した場合は NotifySnsPostStartFailure でカスタムメトリクスを発行し、ImageBatchSucceeded で成功終了する。画像生成 ECS タスクは既に成功しているため、SNS 投稿の起動失敗のみでワークフロー全体を失敗にはしない。

Scheduler アラーム行の追加（Phase 7-1 実装時） 当初の本表には AWS/Scheduler のアラームが未記載だったが、stacks.html セクション 3.4（必須監視対象）とセクション 10（Scheduler DLQ の監視）の要求に合わせて、実装時に 6 メトリクスの個別アラームとして追加した。設定値は Step Functions 失敗アラームと同一とする。
Aurora アラームの DatapointsToAlarm を 2 → 3 に延長（2026-07-22、Phase 13-2） Aurora の CPU / メモリアラームは当初 3 データポイント中 2 で発報としていたが、min ACU 0 からの再開直後の低容量状態（0.5〜1 ACU）で構造的に閾値を超え、バッチ起動のたびに ALARM → 数分で OK 復帰する誤報が発生した（10-4 の DDL 適用時に初観測。定常運用初日の 2026-07-22 12:00 JST の定時実行でもメモリアラームが発報し、1 日 3 回の通知ノイズになることを確認）。 実績上の一時的発報は 1〜6 分で収束するため、3/3（15 分継続）でのみ発報するよう変更した。持続的な高負荷は引き続き検知できる（検知は最大 5〜10 分遅くなるが、本システムはバッチのみで即時対応を要しない）。経緯は設計課題リスト 2026-07-18 の項を参照。

ECS タスク異常終了の監視 ECS タスクの終了コードは CloudWatch の標準メトリクスとして提供されないため、CloudWatch Alarm では直接監視できない。代わりに EventBridge Rule で ECS Task State Change イベントを捕捉し、SNS Topic に通知する。コンテナ起動前に ECS API レベルで失敗したケースは Step Functions 側の ExecutionsFailed アラームで補完する。
