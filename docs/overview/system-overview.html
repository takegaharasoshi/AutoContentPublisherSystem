システム概要

Fix（Phase 10-1 で見直し済み） 最終更新: 2026-07-23

1. システムの目的

最終目標は収益化を狙った SNS 運用である。収益化経路・予算・KPI 等の事業方針は事業戦略書で管理する（2026-07-23 に事業戦略層を新設）。

本システムはその手段として、AWS 上のマネージドサービスを活用し、画像生成から SNS 投稿までを自動化するバッチシステムを提供する。セット（テーマ）の追加・廃止・入れ替えを低コストで繰り返し、試行錯誤で当たりコンテンツを探索するための基盤である。

2. システム構成（概要）

2.1 画像生成バッチ

  1. EventBridge Scheduler が定期的に画像生成ワークフロー（Step Functions）を起動する
  2. DB 準備確認（Aurora の利用可能確認）の後、画像生成 ECS Fargate タスクが画像生成 API で画像を生成する。利用する API・生成方式はセットごとに選択・差し替え可能とし、特定のサービス名に固定しない（方式カタログは app/batch-flow.html セクション 2.1 を参照）
  3. 生成した画像を S3 に保存し、メタ情報を Aurora Serverless v2 に登録する

2.2 SNS 投稿バッチ

  1. 画像生成ワークフローの成功後に SNS 投稿ワークフロー（Step Functions）が自動起動する（手動での単独実行も可能）
  2. DB 準備確認の後、SNS 投稿 ECS Fargate タスクが DB から投稿対象を取得する
  3. S3 の画像に対して Presigned URL を発行し、投稿先プラットフォームの API に渡して投稿する。将来的に複数種類の SNS 媒体へ投稿する前提とし、特定のプラットフォームに固定しない（対応プラットフォーム・投稿フローの詳細は app/batch-flow.html セクション 3 を参照）
  4. 投稿結果を DB に記録する
アプリ仕様の定義場所 投稿対象の決め方・冪等性・再試行などの処理ロジックはアプリ設計（docs/app/）で定義済み（Phase 9 で詳細化・Phase 10-1 で最終 Fix）。本ドキュメントおよびインフラ設計書では実行基盤のみを扱う。

5. 開発方針

  • モノリポジトリを採用し、全サービスを単一リポジトリで管理する
  • 機能単位（サービス単位）でコンテナイメージを分割する
  • CodePipeline / CodeBuild はサービスごとに分ける
  • CDK Deploy はサービスごとに分ける
  • 共通インフラと各サービスは CDK スタックを分離する
  • インフラを先に構築し、業務ロジック（アプリ）は後から設計・実装する（進め方は development-plan.md を参照）

6. 拡張計画

  • 画像生成バッチと SNS 投稿バッチを 1 セットとし、将来的に複数セットを運用する
    • 複数セット運用時は、Step Functions ステートマシンを共有し、EventBridge Scheduler をセットごとに追加する方式を採る
    • スケジュール定義のマスタは IaC（CDK）とする
    • セット間の差分は DB のセット設定と生成方式 strategy モジュール（差し替え可能）で表現し、投稿先 SNS も複数プラットフォーム対応を前提とする（設計書は app/design-outline.html セクション 1.1 の 3 層構造〔契約 / 方式 / セット〕で管理する）
  • 管理画面（オンラインシステム）を追加し、設定・履歴・バッチ管理を GUI で行えるようにする
