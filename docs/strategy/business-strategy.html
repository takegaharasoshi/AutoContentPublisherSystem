事業戦略書

現行版 最終更新: 2026-07-23 初版は 2026-07-23 の壁打ち（Phase 14-1）に基づく
本書の位置づけ 収益化戦略・予算・KPI・プラットフォーム展開方針など、システム設計より上位の事業意思決定の「正」。 システム設計書（docs/infra/・docs/app/）には事業論点を書かず、本書から参照する（体系ガイドのスコープガード）。 開発の進捗管理は従来どおり development-plan.md が担う。

1. 最終目標と基本戦略

  • 最終目標は収益化を狙った SNS 運用（A-1 壁打ち記録から継承）。いきなりの収益化は難しいため、段階を踏む
  • 基本戦略: 全自動運用の仕組みを土台に、セット（テーマ）のスクラップ&ビルドで当たりコンテンツを探索し、当たりを増幅する
  • 段階の整理:
    1. 仕組みの確立 — 完了（2026-07-22 に 1 日 3 回の全自動投稿の定常運用を開始）
    2. リーチ獲得能力の構築 — 動画（リール）対応。収益化経路（セクション 2）の必要条件（Phase 14）
    3. 勝負セットの投入と探索 — リーチを狙うテーマで試行錯誤（Phase 15〜16）
    4. 収益化・増幅 — 当たりの兆しが出たらプラットフォーム展開等で増幅（Phase 17）
優先順位の考え方（2026-07-23 壁打ち） 収益化のボトルネックは配信面の数ではなく「当たりコンテンツの発見」と「リーチ資産」。 プラットフォーム展開は増幅器であり、当たりがない状態で配信面を増やしても運用負荷だけが増える。 したがって「1 つのプラットフォーム（Instagram）で当たりの兆しを掴む → 横展開で増幅」の順とする。

2. 収益化経路

  • 本命: Meta 公式ボーナスプログラム（リール再生数等に応じた報酬）
  • 2026-07 時点の調査による特性（着手時に最新条件を再確認すること）:
    • 招待制（Meta 側から招待されるのを待つ形式。基準は非公開）・期間限定開催
    • 報酬水準の目安は 1 リール再生あたり約 0.03〜0.08 円。月数万円の収益にも 100 万再生規模が必要
  • 含意: この路線は「リールで大きなリーチを取れるアカウントを作る」こととほぼ同義であり、動画（リール）対応が最重要機能となる
リスク: 非オリジナルコンテンツ対策の強化 Meta は転載や「変換の少ない低労力 AI コンテンツ」をリーチ抑制・収益化プログラム除外の対象にする方針を強化している（2026-07 時点の調査）。 全自動 AI 投稿はこの対象に近い領域であるため、以下を必須要件とする:
  • 独自の世界観・スタイルの統一 — 一目でそのアカウントと分かる独自性を勝負セットのテーマ選定要件に含める（Phase 15）
  • AI 利用の開示 — キャプション等での開示方針を動画対応の設計（Phase 14）で具体化する
  • 他者コンテンツの転載・再利用型のコンテンツは扱わない
ボーナス一本足にしない構え KPI はアカウント資産（フォロワー・再生数）に置く。資産があれば現金化手段はボーナス以外（アフィリエイト・グッズ販売等）にも転用できるため、 ボーナスプログラムの制度変更リスクに事業全体が引きずられない。

3. 予算

  • 月額上限 2 万円（AWS 利用料 + 生成 AI API 利用料 + その他固定費の合計）
  • 探索期（収益ゼロの期間）は月額固定費（サブスク・有料 API プラン）を原則積まない。従量課金を基本とする（X API を当面見送る根拠。セクション 5）

費目目安備考
AWS 基盤（Aurora・Fargate・S3 等） 月数千円規模 実績は Cost Explorer で月次確認
画像生成 API 1 日 3 枚で月 1,000〜3,000 円規模 現行 fantasy-animals-1 の運用

5. プラットフォーム展開方針

プラットフォーム位置づけ判断根拠
Instagram 主戦場（現行） 収益化経路（Meta ボーナス）の本命。リール対応で発見面を狙う
TikTok 展開の第一候補（動画資産の獲得後） 動画コンテンツをそのまま転用できる。Content Posting API のアプリ審査等の要件は着手時に調査する
X 当面見送り API の有料化により月額固定費が予算方針（セクション 3）と整合しない。静止画中心で単価も低い
  • 展開の判断基準: 勝負セットで当たりの兆し（M2 到達が目安）が見えてから着手する（セクション 1 の優先順位の考え方）
  • 同一セットの投稿を複数プラットフォームへ展開する構造（セット : アカウント = 1 : N）は設計済み（A-1 壁打ち記録 セクション 2）

7. 更新ルール

  • フェーズ完了時（development-plan.md のフェーズ単位）に本書の前提が変わっていないかを見直す
